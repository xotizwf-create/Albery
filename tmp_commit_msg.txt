feat(fetch_url): deep link reading — browser headers, reader fallback for JS/anti-bot pages, Word/PDF/Excel by URL

Diagnosis (turn #447): the user sent a Dzen article link; the direct fetch got a redirect to
sso.passport.yandex.ru (auth-wall) even with a browser UA — verified by probe from the box. The
reader proxy (r.jina.ai) renders the page and returns the full article text (10.5k chars, verified).
Related gap (#398): the legal agent could not read a .docx by URL — binary docs were decoded as
garbage text.

fetch_url upgrades (parity with the Hermes Brain read-links flow, no heavy browser on the 2GB box):
- real browser User-Agent + Accept-Language headers;
- Word/PDF/Excel by URL: the document is downloaded (cap FETCH_URL_DOC_MAX_BYTES=12MB) and its text
  extracted locally with the same pypdf/python-docx/openpyxl extractors the chat bot uses;
- auth-wall / empty JS-shell detection (login-host redirect or <500 chars after strip) and hard
  4xx/5xx on public pages -> automatic re-read through the reader proxy (FETCH_URL_READER_BASE,
  kill-switch FETCH_URL_READER=0), result marked kind='reader';
- privacy guard: internal/token-bearing hosts (mcp.m4s.ru, *.bitrix24.ru, localhost, private nets;
  FETCH_URL_READER_EXCLUDE) are NEVER sent to the external reader.

Маршрутная карта: new route «пользователь прислал ссылку -> fetch_url» (built-in web tool returning
an auth redirect is not a dead end). Tool description rewritten so agents pick it for links.

Tests: 150 passing (4 new: binary detection, private-host guard, auth-wall detection, docx roundtrip).
