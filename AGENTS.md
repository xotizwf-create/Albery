# Правила безопасной работы с Albery — ОБЯЗАТЕЛЬНЫ для любого агента

Это ПРОД-система: внутри работают живые сотрудники, ошибка видна людям немедленно.
Эти правила действуют для любого агента, работающего с проектом (Claude Code, codex,
Hermes, агент-разработчик и все будущие). Нарушать их нельзя, «маленькая правка» —
не исключение. (Копия этого файла — CLAUDE.md; правишь один — правь оба.)

## 1. Бэкап ДО любого изменения — всегда

- Код: `tar czf /var/backups/albery/code/pre-<задача>-$(date +%Y%m%d_%H%M%S).tar.gz <файлы>`
  плюс точка отката в git (коммит/тег).
- `.env` и конфиги: копия рядом (`env-backup-*`, `config.yaml.bak-<задача>-*`).
- База данных: `pg_dump` затрагиваемых таблиц ДО миграции/массовой правки.
- Нельзя сделать бэкап — не начинай изменение; скажи об этом честно.

## 2. Конвейер кода (никаких правок «сразу на проде»)

Локально: правка → `pyflakes` (0 новых замечаний) → `pytest` (всё зелёное) → push →
CI зелёный. Только затем на 186: `git pull --ff-only` → `python -m py_compile <файлы>` →
рестарт (см. п.3) → `scripts/deploy_smoke.py` → журнал (`journalctl -u albery -p err`) чист.
Живой патч на боксе = сразу коммит и пуш: прод-дерево git всегда чистое.

## 3. Рестарты — только в пустое окно

`systemctl restart albery` убивает живые ходы сотрудников. Рестарт ТОЛЬКО когда
`bitrix_inflight_turns` пуст И нет running-автоматизаций (цикл-ожидание до ~6 минут —
паттерн safe_restart). Без необходимости не рестартовать вообще: новые фичи делай
ОТДЕЛЬНЫМИ модулями/сервисами (пример: albery-tg.service) и добавляй env-тумблеры
для мгновенного отката без деплоя.

## 4. Бокс слабый (2 ГБ RAM) — тяжёлое делать не здесь

Перед любым тяжёлым шагом — `free -m`. Никаких `npm run build`, полных тест-прогонов и
массовых миграций на проде. Фронт собирается ЛОКАЛЬНО, на бокс уходит готовый `dist`
атомарным свапом (`dist` → `dist.bak.prev`), рестарт для статики не нужен.

## 5. Секреты и .env

Секреты НИКОГДА не попадают в git, в вывод команд и в логи. Правки `/var/www/albery/.env`
— только загруженным python-скриптом (шелл-`printf "\n..."` через ssh теряет `\n` и портит
файл), после правки `chmod 600`.

## 6. Разведка и проверка вживую

Перед кодом — живые пробы API на ТЕСТОВЫХ объектах (задача/воронка с пометкой
«ТЕСТ — УДАЛИТЬ»), после — полная зачистка. После каждого деплоя — сквозной смоук
реального сценария, не только «сервис поднялся». Факты не угадывать — проверять.

## 7. Необратимое — только после явного подтверждения владельца

Удаления (задачи, воронки, сделки, поля с данными), массовые изменения, отправка
сообщений/задач живым людям — сначала показать, что именно произойдёт, дождаться
явного «да», и только потом выполнять.

## 8. Интеграции — только самовосстанавливающиеся

Привязки событий, регистрации команд и прочие действия на портале Bitrix НЕ живут вечно
(переустановка приложения, ротация токена). Любой bind/register обязан быть кодом,
который проверяет и чинит себя на старте процесса и на живых событиях. Разовые ручные
действия на портале запрещены.

## 9. После работы

Коммит + пуш (прод и GitHub не расходятся), обновить доки/changelog, тестовые артефакты
удалены, журнал чист. Работа не считается законченной без проверки, что всё живо.

## 10. Проверка ЛОГИКИ перед сдачей (обязательный чек-лист)

Код может быть технически правильным и при этом ломать логику для человека. Перед сдачей
любой фичи/фикса пройди по чек-листу (каждый пункт — реальный прод-инцидент):

1. **От чьего имени?** Кто автор каждого видимого артефакта (комментарий, сообщение,
   задача)? Ответ позванного бота обязан приходить ОТ этого бота, действие «от лица
   сотрудника» — от сотрудника. (Инцидент: ответы шли от технического «ИИ Агент».)
2. **Каждый исход виден.** Пропиши, что увидит человек при успехе / отказе / отсутствии
   доступа / ошибке. МОЛЧАНИЕ — почти всегда нарушение логики: отказ должен быть
   сообщением («У вас нет доступа…»), а не тишиной. (Инцидент: нет доступа → тишина.)
3. **Защита не ест живых людей.** Любой гард/фильтр/скип проверь против легитимных
   пользовательских путей: защита должна КЛЕЙМИТЬ СВОИ артефакты, а не банить чужих
   авторов/каналы скопом. (Инцидент: скип автора 22 съедал упоминания из веб-интерфейса.)
4. **Пограничные случаи.** id=0 — валидное значение (ловушка `x or -1`); время в прошлом;
   вечер/выходной; пустой список; первый запуск с нуля; повторная доставка того же события
   (дедуп). (Инциденты: воронка id=0 «не найдена», дедлайн 19:00 уже прошедшего дня.)
5. **Согласованность с системой.** Новое не должно противоречить принятым правилам:
   доступы/тиринг, модель у всех агентов одна, персоны деловые, форматы дат МСК,
   существующие confirm-гейты. Если меняешь правило — меняй его везде, где оно живёт.
6. **Сквозной прогон глазами пользователя.** Юнит-тесты не заменяют живой сценарий:
   сыграй его целиком на проде (тестовые объекты, потом зачистка) и ПРОЧИТАЙ результат
   так, как его увидит человек — от формулировок до автора и времени.
7. **После фикса спроси: «где ещё такой же паттерн?»** — и проверь соседние места
   (тот же гард, тот же расчёт, та же регистрация) прежде чем закрывать задачу.

## 11. Параллельная работа — ОДНОВРЕМЕННО с тобой может работать другой агент

С тобой в этой же системе может параллельно работать другой агент (Claude Code, codex,
Hermes-разработчик). Прод-дерево на 186 само коммитит и пушит (deploy key +
brain-dirty-watchdog), поэтому origin/main может уйти вперёд, пока ты работаешь. Каждый
пункт ниже — из реального инцидента 17.07.2026 (два агента параллельно; один записал
комментарий в cron-поле строки реестра → упала 500-й вся вкладка «Автоматизации»).

1. **Осмотрись ДО старта.** `git log --oneline -15` + свежие задачи Bitrix: посмотри, что
   другой агент только что менял. Начинай от актуального origin/main, а не от старого клона.
2. **Не доверяй своему снимку состояния.** Файл или строку БД ПЕРЕЧИТАЙ прямо перед правкой —
   другой агент мог изменить их с тех пор, как ты смотрел. Особенно общие горячие файлы
   (`app.py`, `agent_center.py`, `b24bot.py`) и общие таблицы.
3. **Push будет отклонён, если другой агент запушил первым.** `git pull --rebase`, ПРОГОНИ
   заново compile/тесты (его коммиты теперь в твоём дереве), затем push. Никогда не
   force-push поверх чужих коммитов.
4. **Деплой тянет ВСЮ ветку — вместе с чужими коммитами.** После `git pull` на проде
   смоук-тесть НЕ только свой файл, а весь затронутый контур: открой страницы агентов и
   автоматизаций, прогони `deploy_smoke`, глянь журнал. Чужая правка могла сломать общую
   страницу (именно так и случилось 17.07).
5. **Рестарт поднимает изменения ОБОИХ агентов разом.** Рестарт только при пустом
   `bitrix_inflight_turns`; после рестарта проверь и свою фичу, И что ничего соседнее не
   отвалилось.
6. **Запись в ОБЩИЕ реестры — строгая валидация + построчная деградация.** Строка, которую
   ты пишешь в общую таблицу (`agent_automations`, `ai_instructions`, любые читаемые общим
   эндпоинтом), обязана точно соответствовать формату (напр. `schedule` = ровно 5 полей
   cron, пояснения — в отдельное поле `description`). А сериализатор списка обязан
   деградировать ПОСТРОЧНО: одна битая строка выводит предупреждение у себя, а не рушит
   500-й весь список для всех.
7. **Изолируйся.** Предпочитай отдельные модули/файлы/строки БД одновременной правке общего
   горячего файла. Новую фичу — отдельным модулем с env-тумблером (мгновенный откат без
   деплоя), как albery-tg.service.
8. **Координируйся через задачи Bitrix + changelog.** Запиши своё изменение сразу (правило
   инженерного журнала), чтобы другой агент увидел его в общей ленте.
