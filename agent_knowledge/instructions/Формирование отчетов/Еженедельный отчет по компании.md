Инструкция: еженедельный управленческий отчет собственнику по компании (owner weekly).

Назначение: собрать управленческую записку собственнику за рабочую неделю. Это не пересказ задач и не сумма дневных отчетов, а документ для приёмки: идём ли по модели управления, где собственнику принять результат, где не принять, где принять частично, и какие 3–7 решений принять на следующую неделю. Отчёт обязан давать (1) общий вердикт по всей команде, (2) сверку ритма встреч, (3) сверку соответствия регламентам, (4) пофамильную оценку руководителей в формате «что принимать / что не принимать / что частично», (5) несостыковки «созвон ↔ Google-таблица».

Формат, стиль и JSON берутся из этой инструкции (отдельного backend-контракта owner_weekly нет: get_report_contract category_key=owner_weekly возвращает пусто — значит контракт здесь).

## Когда формируется
- Рабочая неделя = понедельник–ПЯТНИЦА. Пятница — рабочий день и день контрольной встречи итогов недели (weekly_review). Период по умолчанию — последняя завершённая рабочая неделя (пн–пт). По пятнице так же собирается созвон и идёт сверка «созвон ↔ таблица» — пятницу из отчёта НЕ исключать. Штатный запуск отчёта — вечер пятницы; если какой-то источник за пятницу ещё не закрыт (например дневной owner-отчёт за пятницу не сохранён) — прямо отметить, какие источники за пятницу не закрыты, и пометить вывод по ним предварительным.
- Не формировать недельный отчёт за период, целиком состоящий из выходных. Суббота и воскресенье в недельный отчёт не входят, если собственник явно не попросил иначе.

## Как читать «О компании» (Google-таблицы и доки)
Все рабочие Google-таблицы и документы зеркалируются в раздел «О компании» с полным текстом (company_drive_sources) и читаются ОТТУДА — заходить в Google вживую не нужно. ВАЖНО: keyword-поиск search_company_knowledge по многословной фразе ненадёжен и часто возвращает пусто, даже когда документ есть. Поэтому: открывать нужные документы ПО ИМЕНИ через list_company_files (просмотр всего списка) + get_company_file(google_file_id|folder_id), а не многословным поиском. search_company_knowledge использовать либо с пустым запросом (просмотр), либо одним коротким отличительным словом. Никогда не делать вывод «таблицы/регламента нет» по одному неудачному keyword-поиску.

## Источники и условия допуска (читать ДО написания текста)
Главное правило: недельный отчёт строится из подготовленных источников, а не из сырых чатов или сырых Zoom-транскриптов.
1. get_report_readiness(date_from=пн, date_to=пт) — один раз, понять готовность по дням: дневные owner-отчёты, Zoom-отчёты, чат руководителей.
2. Дневные owner-отчёты за каждый рабочий день недели (get_owner_reports report_kind=daily) — это основной фактический слой: что сдвинулось, что зависло, риски, адресные рекомендации и обратная связь по ним.
3. Zoom-отчёты за неделю (list_zoom_calls, затем готовый analytical_note каждого созвона). Обязательно — ПЯТНИЧНАЯ контрольная встреча итогов недели: из её JSON берётся блок weekly_review (Наталья и Артур: сделано / не завершено / цели на следующую неделю с весами KPI). Сырую транскрибацию в недельный отчёт не вставлять; при нехватке факта точечно поднять get_zoom_call_transcript. Времена встреч в отчёте приводить в МСК (поля start_time_msk/end_time_msk в источнике могут быть в UTC — при необходимости приводить к МСК, +3).
4. Лист «Задачи недели» — главный второй слой, сверяемый с фактом созвона (задачи, веса/доли KPI, статусы «выполнено/в работе», ожидаемые артефакты). Конкретный источник в «О компании» — Google-таблица «Операционная встреча. Албери 2.0» (path: Google Drive / Артур. Рабочие документы), вкладка «Задачи недели». Открывать её по имени через list_company_files + get_company_file (см. раздел «Как читать О компании»), не многословным keyword-поиском. Если в зеркале видна только часть листов или данные за неделю отсутствуют — прямо отметить «лист Задачи недели не подтянулся/устарел», а не молча опускать раздел.
5. Bitrix (search_tasks) — формальные статусы, сроки, ответственные, просрочки и цифры недели; при comments_human_count>0 читать get_task_comments.
6. Чат руководителей (get_chat_transcript нужного dialog_id за неделю, include_ocr=true) — активность/тишина по дням, договорённости, артефакты-ссылки.
7. get_org_structure — сопоставление людей, ролей и адресатов рекомендаций (только реальные пользователи).
8. Регламенты из раздела «О компании» — открывать по именам документов (list_company_files + get_company_file), не многословным keyword-поиском. Точные имена файлов в зеркале: «Копия Ритм встреч и планерок по модели ИУ 22.05.26» (Google-таблица — нормы встреч для раздела 3); «ООО «АЛБЕРИ» ОРГАНИЗАЦИОННАЯ СТРУКТУРА и МАТРИЦА ФУНКЦИЙ» (матрица решений/функций для раздела 4); «Регламент_отчётности_ООО_Албери»; «Регламент_постановки_задач_ООО_Албери»; «Регламент_фиксации_результата_ООО_Албери»; «_Регламент работы в Битрикс24»; «Операционная встреча. Албери 2.0». По ним строятся разделы «Ритм встреч» и «Соответствие регламентам».
9. Предыдущий недельный owner-отчёт (get_owner_reports report_kind=weekly) — для раздела «Динамика»: что изменилось к лучшему, что повторяется неделя к неделе.
10. list_leader_evaluations / get_recommendation_feedback_context — накопленные оценки руководителей и обратная связь за неделю, если есть.

Если ключевой источник недели отсутствует (нет дневных owner-отчётов, нет отчёта по пятничной контрольной встрече, не подтянулся лист «Задачи недели») — не выдумывать: либо сначала собрать недостающий источник, либо явно пометить раздел как «источник не закрыт, вывод предварительный».

## Правило идентификации людей: транскрибация, а не метаданные
Кто присутствовал и под каким именем — определять ТОЛЬКО по подписям спикеров в транскрибации Zoom (например в расшифровке стоят «Артур», «Наталья»), а НЕ по техническим метаданным: тема созвона, имя Zoom-аккаунта, название «Зал персональной конференции Координатор» — это служебные метаданные аккаунта. Запрещено делать вывод «участник не под своим ФИО» только потому, что в метаданных/теме стоит «Координатор». Нарушение ФИО фиксировать, лишь если в самой транскрибации спикер реально идёт под обезличенной/чужой меткой. По умолчанию, если в расшифровке люди подписаны именами — правило ФИО соблюдено.

## Обязательная структура report_text
Заголовок: «ЕЖЕНЕДЕЛЬНАЯ УПРАВЛЕНЧЕСКАЯ ЗАПИСКА СОБСТВЕННИКУ», период ДД.ММ–ДД.ММ.ГГГГ, при досрочной сборке — пометка о пятнице.

1. Главный вердикт собственнику (по всей команде). Три ответа: (а) идём ли по модели управления — да / частично / нет, одним абзацем с обоснованием; (б) главное отклонение недели; (в) что решить на этой неделе одним ключевым решением. Сразу после — короткие списки «Что можно принять» и «Что нельзя принимать полностью / только частично» по итогам недели. Это верх записки: без технических id, без MCP/коннекторов.

2. Динамика vs прошлая неделя + цифры недели. Что улучшилось и что повторяется по сравнению с предыдущим недельным отчётом. Блок «Цифры недели»: задачи (новых / открыто / просрочено, % просрочки от открытых); Zoom (созвонов / суммарно минут / сколько из 5 дней покрыто); чат руководителей (сообщений, в какие дни тишина). Цифры брать из Bitrix, list_zoom_calls и транскрипта чата руководителей, не выдумывать. ПРОВЕРКА СВЕЖЕСТИ BITRIX (важно): если за неделю НЕТ новых задач или появление задач прекратилось — это, скорее всего, ИСТЁКШАЯ ПОДПИСКА BITRIX MARKETPLACE (REST-интеграция отрезана → задачи перестали подтягиваться), а НЕ реальное отсутствие работы. Тогда НЕ подавать старые числа («открыто 78» и т.п.) как актуальные: пометить «⚠️ данные Bitrix устарели — вероятно истекла подписка Bitrix Marketplace», вынести отдельным риском и добавить в раздел 9 решение собственнику «проверить/продлить подписку Bitrix Marketplace». Всегда сверяться с актуальными данными, не выдавать устаревшие за свежие.

3. Ритм встреч за неделю. Оформи РОВНО в виде markdown-таблицы по дням пн–пт (время МСК; контрольная встреча итогов недели — в пятницу), ОБЯЗАТЕЛЬНО со строкой-разделителем | --- | (без неё таблица не отрендерится в PDF), строго вида:
| День | Норма по регламенту | Факт (созвон, время, длит., тема) | Статус |
| --- | --- | --- | --- |
| Пн ДД.ММ | … | … | ✅/⚠️/❌ |
Строки на каждый день Пн–Пт (пятница — рабочий день с контрольной встречей итогов недели). Статус: ✅ норма / ⚠️ частично / ❌ нет. Под таблицей — вывод по ритму: сколько дней подтверждено по факту наличия Zoom, и где ритм слабее регламента (короткие встречи, неполный состав, нет содержательной расшифровки). Нормы встреч — из «Ритм встреч и планёрок по модели ИУ»; две несведённые версии времени контрольной встречи отмечать явно. Имена и состав участников — по транскрибации (см. правило идентификации людей выше), метаданные «Координатор» нарушением не считать.

4. Соответствие регламентам. Таблица: регламент × норма × факт недели × статус × что сделать. Минимальный обязательный набор строк: матрица решений (кто инициирует/согласует/утверждает/исполняет; что уходит на утверждение Евгению — прибыль, финмодель, лимиты, договоры, оргструктура, масштаб); присутствие под своим ФИО в управленческих Zoom (проверять ПО ТРАНСКРИБАЦИИ, не по метаданным/теме созвона); веса KPI на задачах недели; календарь/регламент отчётности; регламент постановки задач (срок, ожидаемый результат, формат результата, кто принимает); регламент фиксации результата («выполнено» только с проверяемым артефактом). В конце раздела — источники-регламенты по названиям документов из «О компании». Вывод: на каких 2–3 местах модель рвётся и почему.

5. Поведенческие факторы. По каждому значимому участнику (Наталья, Артур, Дмитрий, Оксана, Евгений и др., кто проявился): что реально происходило + эпизод-доказательство (созвон ДД.ММ, таймкод/короткая цитата) + риск для дела. Без пустых ярлыков «проявлял инициативность» без эпизода. Если по человеку нет значимого сигнала — так и написать.

6. Оценка руководителей: Наталья Горюнова и Артур Степанян (ГЛАВНЫЙ раздел, формат строгий — ТАБЛИЦАМИ по каждому руководителю, так удобнее читать). По каждому руководителю — отдельная markdown-ТАБЛИЦА, ОБЯЗАТЕЛЬНО со строкой-разделителем | --- | (без неё таблица не отрендерится в PDF), строго вида:
| Задача (план + вес KPI) | Факт (Google-таблица ↔ созвон, таймкод) | Вердикт собственника (✅ принять / ⚠️ частично / 🚫 заблокировано / ❌ не принимать) | Требуемый артефакт / следующий шаг |
| --- | --- | --- | --- |
| … | … | ✅/⚠️/🚫/❌ | … | Одна строка — на КАЖДУЮ задачу руководителя из листа «Задачи недели», ничего не сворачивать. Логика заполнения ячеек:
   - Факт: что стоит в Google-таблице (статус, вес, артефакт) И что подтверждено/опровергнуто на созвоне (таймкод).
   - Решение собственника: принимать как выполнено / принимать частично / не принимать как выполнено / принимать как старт-попытку-эскалацию. Если в таблице «выполнено», а на созвоне «в работе / частично / заблокировано решением Евгения» — НЕЛЬЗЯ принимать как выполнено, корректный статус «частично / заблокировано».
   - Требуемый артефакт/следующий шаг: что именно собственник должен получить, чтобы принять задачу.
   Начинать с коммерческого/проектного факта (например план заказов: план N шт / факт M шт). Каждую задачу из листа «Задачи недели» разобрать отдельно, ничего не сворачивать. Базовый принцип: «выполнено» без проверяемого артефакта и при открытом блокере не принимается.

7. Несостыковки «созвон ↔ Google-таблица». Перечислить конкретные расхождения: где в таблице «выполнено» при незакрытом блокере; где артефакты у одного руководителя сильнее, чем у другого; где план следующей недели перенесён, но без точных дат; где в таблице появились задачи, не обсуждённые на пятничном созвоне. По каждой — управленческий вывод.

8. План на следующую неделю: сверка с обсуждением. По Наталье и Артуру: что стоит в таблице (с весами %) против того, что обсуждали на созвоне; оценка совпадения и чего не хватает (явные задачи, резерв времени, контрольные даты внутри недели). Для каждой задачи требовать не только вес, но и день проверки.

9. Что собственнику сделать сейчас / Решения собственника на N рабочих дней. Нумерованная таблица: решение (действие) × ответственный (реальный человек-исполнитель) × срок (конкретная дата). ВАЖНО про ответственного: собственника (Александра Никитенко) НЕЛЬЗЯ ставить ответственным — он принимает решения, но не исполняет их, и он же не «внедряет» операционные процессы. Исполнители решений — это руководители и ген. директор: Евгений Палей (ген. директор, владелец управленческой модели и арбитр по матрице решений — прибыль, финмодель, договоры, оргструктура), Наталья Горюнова, Артур Степанян, Дмитрий Строгонов. Если решение по сути — это акт приёмки самим собственником (например «принять коммерческий результат»), ответственным за фиксацию статуса ставить Евгения Палея, а не Александра. Включить правило приёмки: «выполнено» ставится только при наличии артефакта и снятых блокерах.

10. Итоговая оценка недели. 3–5 строк: управленческий итог, по кому что принимать, и одна главная задача следующей недели.

## Четыре поля усиления (сквозное требование к разделам 6, 8, 9)
Любая управленческая задача руководителя для приёмки должна иметь четыре поля: точная дата проверки, проверяемый артефакт, кто утверждает (часто Евгений по матрице решений), критерий приёмки. Если поля нет — прямо указать пробел.

## Что запрещено
- Технические id, номера call_id/dialog_id, упоминания MCP, коннекторов, пайплайна, названий инструментов в видимом тексте.
- Принимать статус «выполнено» из таблицы без проверки по созвону и без артефакта.
- Сворачивать задачи руководителей в общие формулировки или пропускать задачу из листа «Задачи недели».
- Заменять пофамильную оценку руководителей общим абзацем — раздел 6 обязателен и детальный.
- Заменять таблицы разделов 3, 4, 6 и 9 прозой или маркированным списком — эти разделы оформляются ТОЛЬКО markdown-таблицей со строкой-разделителем | --- |; отчёт без этих таблиц считается бракованным и подлежит переделке.
- Ставить собственника (Александра Никитенко) ответственным в таблице решений; адресовать решения ролям без человека («Координатор», «административный блок»).
- Считать нарушением ФИО техническое имя Zoom-аккаунта/тему созвона («Координатор») — идентификация людей только по транскрибации.
- Делать вывод «документа/таблицы нет в О компании» по одному неудачному многословному keyword-поиску — сначала просмотреть список через list_company_files и открыть по имени.
- Строить отчёт только по таблице или только по созвону — обязательно сверять оба слоя.

## Сохранение
Сохранять только после явного «сохраняй/ставь/да» собственника, если запуск не штатно-автоматический. Сохранение через save_owner_weekly_report (period_start, period_end, status=done). Заполнять:
- report_text — полный текст по структуре выше;
- summary — главный вердикт в 2–3 строки;
- dynamics_summary — раздел 2; risks_summary — ключевые риски; recommendations — краткий список решений собственника;
- weekly_dynamics, goal_dynamics — структурно по руководителям (сделано/не завершено/цели с весами KPI из weekly_review пятничного созвона);
- overdue_tasks, open_tasks — из Bitrix; manager_messages, manager_recommendations — если в отчёте есть адресные рекомендации (тогда действует правило двойного сохранения рекомендаций из инструкции «Адресные рекомендации и обратная связь»);
- raw_json — манифест источников (какие дневные отчёты, Zoom, регламенты, лист «Задачи недели» использованы).
