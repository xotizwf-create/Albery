Инструкция: ежедневный отчет по компании.

Назначение: сформировать ежедневный управленческий отчет для собственника по компании. Это не техническая сводка и не каталог задач. Итог должен отвечать на вопросы: что сдвинулось, что зависло, какие риски, кому какой адресный вопрос нужно отправить.

## Календарное правило: только рабочие дни

Ежедневный отчет по компании, связанные daily-отчеты-источники и адресные рекомендации формируются только за рабочие дни.

Запрещено формировать или отправлять отчеты и рекомендации за субботу и воскресенье, если собственник отдельно и явно не попросил сформировать отчет именно за выходной день.

Правило перехода между рабочими днями:
- в пятницу формируется пятничный отчет;
- в субботу и воскресенье ежедневные отчеты и рекомендации не формируются;
- в понедельник отчет формируется на основании пятничного отчета как предыдущего рабочего дня, а не на основании субботы или воскресенья;
- если для понедельника нужен предыдущий owner-отчет, использовать последнюю доступную рабочую дату, обычно пятницу.

Если автоматизация/cron запущена в выходной день, нужно остановиться без формирования отчета, без сохранения no_data и без отправки рекомендаций.

Главное правило: отчет по компании формируется только из подготовленных отчетов-источников, а не из сырых чатов, сырых Zoom-транскриптов или одной выгрузки Bitrix.

Перед началом обязательно открой раздел:
Сводная аналитика -> Настройка промтов -> ежедневный общий отчет для собственника.
Активный ИИ-промт из этого раздела является основным контрактом структуры, стиля и JSON-ответа. Эта инструкция задает порядок подготовки источников и запреты.

Условия допуска к формированию:
1. Сформированы daily-отчеты только по активным чатам, где за report_date есть сообщения, и только если report_date является рабочим днем. Активные чаты без сообщений за дату игнорируются: для них не создается no_data-отчет и они не блокируют owner-отчет.
2. Сформированы аналитические отчеты по каждому Zoom-созвону за report_date, если report_date является рабочим днем.
3. Сформирован предыдущий ежедневный отчет для собственника за предыдущий рабочий день. Для понедельника предыдущий рабочий день — пятница.

Если хотя бы одно условие не выполнено:
- не формируй ежедневный отчет по компании;
- сначала сформируй недостающие отчеты по инструкциям соответствующего типа;
- для чата используй инструкцию ежедневного отчета по чату;
- для Zoom используй инструкцию отчета по Zoom-созвону;
- для предыдущего owner-отчета сначала сформируй отчет за предыдущую доступную рабочую дату;
- после подготовки источников вернись к ежедневному отчету по компании.

Что читать при формировании:
1. Каждый daily-отчет каждого активного чата, где за дату есть сообщения. Общей сводки по всем чатам как источника нет: нельзя опираться на chat_overall_daily_report вместо чтения отчетов конкретных чатов.
2. Каждый Zoom-отчет за дату. Используй только готовый аналитический отчет Zoom, без текста сырой транскрибации.
3. Предыдущий ежедневный отчет собственнику за предыдущий рабочий день целиком, включая адресные рекомендации. Если в текущих отчетах по чатам, Zoom или Bitrix нет явного подтверждения выполнения прошлой адресной рекомендации, повтори ее в новом пункте 6.
4. Bitrix используй как формальный источник статусов, сроков, ответственных и просрочек.
5. Оргструктуру используй для проверки адресатов: адресные рекомендации можно давать только реальным пользователям Bitrix.
6. Обратную связь сотрудников по прошлым адресным рекомендациям: ответы, согласие, возражения, делегирование, странные/уклончивые ответы и отсутствие ответа.
7. Регламенты компании из раздела "О компании": роли, владельцы процессов, порядок согласований, платежный календарь, ритм встреч, матрицы решений и SLA.

Что запрещено:
- формировать или отправлять ежедневный отчет/рекомендации в выходные без явного отдельного указания собственника;
- формировать отчет по одному чату, если за день есть несколько активных чатов;
- использовать общую сводку по всем чатам вместо отчетов каждого конкретного чата;
- использовать сырую Zoom-транскрибацию в ежедневном отчете по компании;
- писать техническую сводку вида "найдено N чатов / N задач";
- упоминать MCP, коннекторы, пайплайн, get_period_index, search_tasks;
- строить отчет вокруг номеров Bitrix, call_id или dialog_id;
- адресовать рекомендации ролям без пользователя: "Координатор", "ответственный", "юридический контур", "административный блок".

Требования к качеству:
- отчет должен быть управленческим, как записка собственнику;
- верхние блоки должны быть без технических id;
- в полном тексте допускаются ссылки на источник только когда они нужны для проверки факта;
- пункт 6 обязателен и должен содержать готовые адресные сообщения реальным людям из оргструктуры;
- старые незакрытые адресные вопросы из предыдущего owner-отчета не заменяются новыми темами по тому же человеку, а повторяются до явного подтверждения закрытия.
- каждую адресную рекомендацию начинай с мягкого вступления, которое учитывает реакцию адресата на предыдущую рекомендацию: поблагодари за ответ, отметь принятое замечание, аккуратно продолжи новым предложением. Если ответа не было, начни нейтрально и попроси коротко подтвердить статус.
- каждую рекомендацию сверяй с регламентами: если фактический исполнитель, владелец решения, срок или процесс расходится с регламентом, прямо укажи это в сообщении и предложи корректное действие по регламенту. Например: если регламент закрепляет платежный календарь за Анастасией, а фактически его ведет Артур, в рекомендации Артуру нужно мягко указать на регламент и предложить делегировать/согласовать передачу Анастасии.
