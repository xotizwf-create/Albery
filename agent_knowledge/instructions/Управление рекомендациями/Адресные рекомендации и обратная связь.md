# Адресные рекомендации и обратная связь

Эта инструкция описывает полный жизненный цикл адресных рекомендаций: создание, постановка в Bitrix24 задачами, получение ответа, ИИ-интерпретация, статус и управленческая проверка.

## Главный принцип

Адресная рекомендация — отдельная сущность, а не просто текст в отчете или сообщении. У нее должен быть `recommendation_id`, адресат, источник, статус и журнал событий.

Используемые таблицы:
- `owner_manager_recommendations` — текущая карточка рекомендации и ее актуальный статус;
- `owner_recommendation_dispatches` — факты постановки в Bitrix24 или другой канал;
- `owner_recommendation_events` — неизменяемый журнал событий: постановка в Bitrix24, ответы, интерпретации, смена статусов, проверки руководителем.

## Канал исполнения рекомендаций: только задачи Bitrix24, не личные сообщения

Актуальное правило после обновления: все адресные рекомендации из ежедневных отчетов собственнику нужно ставить в Bitrix24 как задачи. Личные сообщения больше не являются стандартным каналом доставки рекомендаций.

По умолчанию запрещено:
- отправлять рекомендации из ежедневного owner report через `send_owner_recommendations_to_bitrix`;
- отправлять рекомендации через `send_bitrix_message`;
- считать личное сообщение полноценной постановкой рекомендации;
- писать сотруднику в ЛС вместо создания задачи, если собственник явно не попросил именно ЛС.

Стандартная механика для ежедневного отчета:
1. Сформировать адресные рекомендации в отчете.
2. Создать/проверить отдельные карточки `owner_manager_recommendations`.
3. Для каждого адресата сгруппировать все его рекомендации за дату в одну Bitrix24-задачу.
4. Название задачи: `Рекомендации ДД.ММ`, например `Рекомендации 01.06`.
5. Ответственный: сотрудник, которому адресованы рекомендации.
6. Постановка: вечером примерно в 18:00 МСК после подтверждения собственника/штатного запуска.
7. Дедлайн: до 10:00 МСК следующего дня.
8. Срок задачи нельзя переносить. Это должно быть явно указано в описании задачи.
9. В описание задачи вставляется стандартный блок реакции и затем нумерованный список рекомендаций.
10. Записать факт постановки в `owner_recommendation_dispatches` и событие `owner_recommendation_events` со статусом `task_created` / `sent`.
11. Дальше отслеживать обратную связь и выполнение через Bitrix-задачу/комментарии, релевантные чаты и Zoom.

Стандартный текст задачи:

```text
Ознакомьтесь со списком рекомендаций и дайте на них реакцию:
- неактуально
- взял в работу
- уже в работе
- уже есть результат
- уже есть рабочий файл — "Ссылка".

Срок этой задачи переносить нельзя.

Текст рекомендаций:
1. ...
2. ...
3. ...
```

Если рекомендаций для одного адресата несколько, запрещено создавать отдельную задачу на каждую рекомендацию без отдельного указания собственника. Нужна одна агрегированная задача `Рекомендации ДД.ММ` на адресата со всеми пунктами.

Если инструмент постановки в Bitrix24 требует подтверждения, перед созданием нужно показать собственнику: название, ответственного, дедлайн, наблюдателей, текст/описание и только после явного «ставь/создавай/да» выполнять действие.

## Обязательное правило сохранения owner daily/weekly report

При сохранении ежедневного или недельного отчета собственнику адресные рекомендации должны быть сохранены в двух местах одновременно:

1. В `report_text` — полный раздел «Адресные рекомендации» с группировкой по адресатам, приоритетом, сроком и готовым текстом/содержанием рекомендации.
2. В структурированных JSON-полях отчета, из которых backend создает строки `owner_manager_recommendations`:
   - `manager_recommendations` — структурированные действия по каждому адресату;
   - `manager_messages` — готовые финальные тексты/описания для постановки задач в Bitrix24;
   - `recommendations` — краткий список ключевых рекомендаций для собственника.

Запрещено сохранять адресные рекомендации только в `recommendations` или только в `report_text`. Это ошибка: интерфейс может показать текст, но механика постановки задач в Bitrix24 и отслеживания не получит отдельные карточки рекомендаций.

После каждого `save_owner_daily_report` или `save_owner_weekly_report`, если в отчете есть адресные рекомендации, ИИ обязан сразу выполнить проверку:

- вызвать `list_pending_owner_recommendations(report_date)` для ежедневного отчета;
- убедиться, что количество и адресаты рекомендаций совпадают с разделом `report_text`;
- проверить, что у каждой рекомендации есть `id`, `manager_full_name`, `manager_bitrix_user_id`, `recommendation_text`, `subject`, `priority`, `due_date`, `status`;
- статус новых непоставленных рекомендаций должен быть `new`, `draft` или `queued`, но не `sent`, `done`, `accepted` или `no_response` без фактического события;
- если список pending пустой или не совпадает с текстом отчета, нужно немедленно пересохранить отчет с корректно заполненными `manager_recommendations` и `manager_messages`, затем повторить проверку.

Отчет считается корректно сохраненным только когда одновременно выполнены условия:
- новая версия отчета стала `is_current = true`;
- полный адресный блок виден в `report_text`;
- `list_pending_owner_recommendations` возвращает отдельные карточки рекомендаций для всех адресатов, которые должны быть поставлены задачами в Bitrix24 и отслежены.

Нельзя сообщать собственнику, что отчет исправлен или готов к постановке в Bitrix24, пока эта проверка не пройдена.

## Важное правило против ложного `no_response`

Запрещено делать вывод `нет обратной связи` только потому, что:
- `list_recommendations` не нашел карточку;
- в групповых чатах 0 сообщений;
- поиск по одному ключевому слову ничего не нашел;
- рекомендация есть только в owner daily report, но еще не заведена как карточка.

Если карточки рекомендации нет, ИИ обязан:
1. Прочитать owner daily/weekly report, где рекомендация была сформулирована.
2. Извлечь адресата, текст рекомендации, дату постановки/ожидаемой реакции, ключевые слова.
3. Через `get_org_structure` уточнить ФИО, возможные варианты имени и Bitrix user id адресата.
4. Проверить Bitrix-задачи/карточки по теме рекомендации и ответственному.
5. Через `list_chats(date_from, date_to, query=ФИО/имя/фамилия)` найти личный диалог и связанные чаты.
6. Если `list_chats` не нашел личный диалог, выполнить `search_messages` глобально по периоду без `dialog_id` по ФИО, имени, фамилии и ключевым словам рекомендации.
7. Если найден диалог или сообщение, обязательно прочитать `get_chat_transcript` за период после постановки рекомендации.
8. Только после чтения релевантной Bitrix-задачи/комментариев/личного диалога/сообщений можно ставить статус по ответу.

Если личный диалог или Bitrix-источник обратной связи не найден или не синхронизирован, писать не `no_response`, а: `источник обратной связи не найден в доступных данных; статус не подтвержден`.

## Где искать ответ

Ответ может быть не в том месте, где сформирована рекомендация. Обязательно проверять:
- Bitrix-задачу `Рекомендации ДД.ММ`, куда поставлена рекомендация;
- комментарии Bitrix-задачи;
- проектный чат по теме рекомендации;
- общий/управленческий чат;
- личный диалог адресата;
- Zoom, если ответ мог быть устным.

Групповые чаты с нулем сообщений не доказывают отсутствие ответа в Bitrix-задаче или личном диалоге.

## Статусы

Использовать статусы:
- `new` — создана из отчета, еще не поставлена задачей в Bitrix24;
- `draft` — подготовлена вручную;
- `queued` — поставлена в очередь постановки;
- `sent` — поставлена/отправлена в рабочий канал;
- `task_created` — создана Bitrix24-задача по рекомендации;
- `seen` — увидена, если канал позволяет;
- `accepted` / `acked` — человек принял к сведению;
- `in_progress` — взял в работу;
- `needs_clarification` — просит уточнение;
- `disagreed` — не согласен;
- `delegated` — передал другому или указал другого ответственного;
- `done` — есть явный факт выполнения;
- `rejected` — отклонено управленчески;
- `no_response` — нет ответа в срок, но только после проверки Bitrix-задачи, личного диалога, релевантных чатов/задач;
- `overdue` — срок ответа/исполнения прошел;
- `requires_manager_review` — ответ неоднозначный, конфликтный или странный;
- `cancelled` — отменено;
- `error` — ошибка постановки/обработки.

## Журнал событий

Каждый ответ и каждая ИИ-интерпретация должны сохраняться отдельным событием `owner_recommendation_events`.

Событие должно содержать:
- `recommendation_id`;
- `event_type`: `sent`, `employee_replied`, `ai_interpreted`, `status_changed`, `manager_reviewed`, `task_created`, `closed`, `source_found` и т.д.;
- `author_type`: `system`, `ai`, `manager`, `employee`;
- `dialog_id`, `bitrix_message_id`, `chat_message_day`, `bitrix_task_id`, если источник — чат или Bitrix;
- `old_status`, `new_status`;
- `event_text`;
- `interpretation` — структурированный разбор ответа;
- `source_payload` — ссылки на отчет, чат, задачу, Zoom или сообщение.

## Как искать обратную связь

Перед анализом обратной связи ИИ должен:
1. Получить активные рекомендации через `list_recommendations(status="open")` или `get_recommendation_feedback_context(dialog_id, report_date)`.
2. Если карточки нет, прочитать owner daily/weekly report и извлечь рекомендацию оттуда.
3. Проверить Bitrix-задачи/карточки по адресату, теме и сроку; если есть human comments — прочитать комментарии.
4. Получить список чатов через `list_chats(date_from, date_to)` и отдельно через `list_chats(query=адресат)`.
5. Для конкретного дневного отчета читать два дня: предыдущий рабочий день и текущий день, а если рекомендация поставлена вечером — расширить окно до следующего дня.
6. Искать ответ по `recommendation_id`, имени адресата, тексту рекомендации, теме, связанному `dialog_id` и `bitrix_task_id`.
7. Если ответ найден, сохранить событие через `save_recommendation_event`.

## Странная обратная связь

Если ответ на рекомендацию не дает понятного управленческого статуса, ИИ обязан пометить его как `Странная обратная связь`.

Примеры:
- "ок" без действия и срока;
- "посмотрю" без обязательства;
- "не понял" без конкретного вопроса;
- ответ не по теме;
- человек соглашается, но пишет факт, противоречащий источникам;
- делегирует без нового владельца;
- пишет, что сделано, но нет подтверждения результата;
- отвечает эмоционально/уклончиво и статус непонятен.

В этом случае:
- `new_status` = `requires_manager_review` или `needs_clarification`;
- `manager_review_required` = true;
- событие сохраняется в `owner_recommendation_events`;
- в дневном отчете по чату должен быть раздел `Странная обратная связь`.

## Связь с ежедневным отчетом по чату

Дневной отчет по чату должен использовать блок `Адресные рекомендации и обратная связь` и возвращать:
- `recommendation_feedback` — понятная реакция;
- `strange_feedback` — неясная/странная реакция.

Backend-пайплайн автоматически сохраняет эти элементы как события рекомендаций. MCP-агент должен делать то же через `save_recommendation_event`, если отчет формируется вручную через MCP.

## Ограничение

ИИ не должен считать рекомендацию выполненной без явного результата. Факт "принял" — это не `done`, а `accepted` или `in_progress` в зависимости от текста.
