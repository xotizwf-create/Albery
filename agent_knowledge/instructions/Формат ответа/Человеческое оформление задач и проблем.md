# Человеческое оформление задач, проблем и Bitrix-данных

Эта инструкция обязательна для любых ответов, где используются Bitrix-задачи, просрочки, проблемы, ответственные, сроки или статусы.

## Главное правило
Ответ должен быть понятен руководителю без знания Bitrix ID.

Bitrix ID - это технический источник, а не смысл ответа. Нельзя строить видимый отчет вокруг ID задач.

## Что запрещено
Запрещено начинать пункт с ID:
- плохо: `317723 — Подготовить список вопросов...`
- плохо: `317889 — Подписать документы...`
- плохо: `Задача 317653 просрочена`.

Запрещено выдавать длинный список задач без группировки.

Запрещено писать только `pending`, `responsible_name = null`, `deadline = null`. В человекочитаемом тексте использовать нормальные формулировки:
- `статус: ждет выполнения`;
- `ответственный не назначен`;
- `срок не указан`.

## Как писать задачу
Правильный формат одной задачи:

`Подготовить список вопросов и согласований для генерального директора`

- `Ответственный:` Анастасия Андрусяк.
- `Статус:` ждет выполнения.
- `Срок:` 11.05.2026 09:00, просрочено.
- `Проблема:` управленческие согласования не закрыты в срок.
- `Источник:` Bitrix, ID 317723.

ID допускается только в поле `Источник` или в скобках в конце строки.

## Как писать проблему по группе задач
Если задач несколько и они относятся к одной проблеме, группируй их.

Правильный формат:

`Финансово-документальные хвосты не закрыты в срок.`

- `Что видно:` несколько задач по закрывающим документам и платежам остались в статусе `ждет выполнения` после дедлайна.
- `Ответственные:` Анастасия Андрусяк, Артур Степанян, Наталья Горюнова, Марианна Харабардина.
- `Сроки:` 11-14 мая 2026.
- `Риск:` финансовый и документальный контур закрывается формально или с задержкой.
- `Примеры источников:` `Предоставить закрывающие документы Карабулут` (Bitrix ID 317681), `Оплатить счета за услуги интернета` (Bitrix ID 317929).

Не перечисляй все 20 задач, если достаточно 2-4 примеров для подтверждения проблемы.

## Приоритет смысла над техническими полями
В каждом пункте сначала управленческий смысл:
1. Что за проблема.
2. Почему это важно.
3. Кто отвечает.
4. Срок и статус.
5. Источник.

Не наоборот.

## Как писать задачи без ответственного
Плохо:
`317867 / 317871 / 317869 — responsible_name = null`.

Хорошо:
`Логистические задачи по контролю прибытия грузов заведены без ответственного.`

- `Что видно:` несколько активных задач по грузам Адонис/Тапаз имеют высокий приоритет, но ответственный не назначен.
- `Статус:` активные / ждут выполнения.
- `Срок:` 20.05.2026.
- `Риск:` если груз задержится или придет с отклонением, непонятно, кто должен реагировать.
- `Источник:` Bitrix-задачи по контролю прибытия грузов на ФФ 356О/357О/362.

## Как писать задачи-заглушки
Если задача называется `Ждем контроля`, `Проверить`, `Актуализировать`, `Обсудить`, не воспринимать это как нормальную задачу.

Формат:
`В Bitrix есть задачи-заглушки без управляемого результата.`

- `Что видно:` несколько задач с названием `Ждем контроля` висят в статусе `ждет выполнения`.
- `Ответственный:` Евгений Палей.
- `Срок:` не указан.
- `Проблема:` по ним невозможно понять объект контроля, критерий закрытия и ожидаемый результат.
- `Что сделать:` закрыть как мусорные или переоформить с объектом контроля, сроком и критерием результата.

## Как писать просрочки
Не нужно перечислять все просрочки подряд. Сначала сгруппируй:
- финансовые/документальные;
- юридические/кадровые;
- логистика/операционка;
- управленческие согласования;
- доступы/администрирование.

Внутри группы покажи 2-5 самых важных примеров.

## Как писать статус
Статус должен быть понятен:
- `ждет выполнения`;
- `в работе`;
- `завершено`;
- `просрочено`;
- `срок не указан`;
- `ответственный не назначен`.

Если источник использует технический статус (`pending`, `completed`), переводи его в человеческий вид, а технический статус можно не показывать.

## Итоговый блок действий
В конце ответа по проблемам должен быть блок `Что сделать первым`.

Формат:
1. `Назначить владельцев на задачи без ответственного` - какие группы задач, кто должен назначить, срок контроля.
2. `Закрыть/переоформить задачи-заглушки` - какие задачи, критерий результата.
3. `Разбить большие задачи на управляемые результаты` - какая задача, какие подзадачи.
4. `Проверить просрочки по конкретным ответственным` - ФИО, группы задач.

## Короткий пример хорошего ответа
`За 6-15 мая видно 4 управленческие проблемы:`

1. `Финансово-документальные задачи зависли после дедлайна.`
   Ответственные: Анастасия Андрусяк, Артур Степанян, Наталья Горюнова. Статус: часть задач ждет выполнения после срока 11-14 мая. Риск: документы и платежи закрываются с задержкой. Источник: Bitrix.

2. `Логистические контрольные задачи заведены без владельца.`
   Ответственный: не назначен. Статус: активные задачи с высоким приоритетом, срок 20 мая. Риск: нет персональной реакции при отклонении по грузу. Источник: Bitrix.

3. `В Bitrix есть задачи-заглушки без результата.`
   Ответственный: Евгений Палей. Статус: ждет выполнения, срок не указан. Пример: `Ждем контроля`. Риск: шум и невозможность контроля исполнения.

Такой формат лучше, чем список ID, потому что показывает проблему, владельца, статус, срок и управленческий риск.
