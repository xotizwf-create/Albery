# Описание доступных инструментов MCP и порядок поиска

Эта инструкция объясняет, какие источники доступны ИИ через MCP-сервер и как быстро искать информацию. Перед незнакомым запросом сначала вызови `get_context_guide`, чтобы получить актуальную карту источников.

## Главный принцип
Не отвечай по памяти, если вопрос относится к компании, сотрудникам, задачам, чатам, Zoom, регламентам или отчетам. Сначала найди данные в MCP, затем отдели факты источников от своих выводов.

Если данных нет в одном источнике, это не значит, что факта нет вообще. Проверь индекс/список источников и смежные инструменты.

## Быстрая карта инструментов

### Состояние MCP и навигация
- `health` - проверить подключение к PostgreSQL и статус MCP.
- `get_context_guide` - получить актуальные правила навигации по источникам. Использовать первым для новых типов задач.
- `list_available_sources` - посмотреть, какие таблицы есть и сколько в них строк. Полезно, если непонятно, есть ли данные вообще.
- `get_compact_export` - получить компактный срез данных за период: задачи, чаты, Zoom и контекст. Использовать для широкой аналитики за период.

### Инструкции и промпты ИИ
- `get_ai_instructions` - прочитать живые инструкции из раздела `Настройки -> ИИ-инструкции`.
- `get_report_contract` - прочитать активные промпты. Обязательно использовать перед генерацией отчетов.

Правило:
- для ежедневного отчета по чату: `get_report_contract(category_key = "chat_analysis")`;
- для Zoom-отчета: `get_report_contract(category_key = "zoom_processing")`.

### О компании: регламенты, документы, процессы, правила
- `search_company_knowledge` - быстрый поиск по разделу `О компании`, включая Google Drive документы. Использовать первым для вопросов о правилах, процессах, регламентах, оргмодели, финансовых моделях и внутренних документах.
- `list_company_files` - получить список всех файлов и папок из `О компании`. Использовать, когда нужно понять, какие документы существуют, или когда поиск по ключевому слову мог не сработать.
- `get_company_file` - прочитать полный текст конкретного документа по `folder_id` или `google_file_id`.
- `get_company_profile` - прочитать общий профиль компании и дерево папок. Использовать для полного обзора раздела.

Быстрый паттерн:
1. `search_company_knowledge(query)`.
2. Если результат неполный: `list_company_files(limit=500)`.
3. По нужному документу: `get_company_file(folder_id)`.
4. В ответе указать источник: название документа, путь, Google Drive URL при наличии.

### Оргструктура и люди
- `get_org_structure(include_inactive=false)` - сотрудники, роли, руководители, отделы, Bitrix user id.

Использовать всегда, когда запрос связан с людьми: кто за что отвечает, кому принадлежит задача, кто руководитель, кто упоминался в Zoom/чате. Если имя неоднозначное, прямо писать `требуется уточнение` и показывать варианты.

### Bitrix-задачи
- `search_tasks` - поиск задач по id, периоду, тексту и ответственному.
- `get_task_comments` - прочитать комментарии/результаты по задаче, если нужно понять ход переписки или проверить качество закрытия.
- `create_bitrix_task` - создать новую задачу, когда известны название, один точный ответственный и дедлайн.
- `add_bitrix_task_comment` - добавить комментарий в существующую задачу; использовать только после определения точного `bitrix_task_id`.
- `reopen_bitrix_task` - возобновить закрытую задачу и оставить комментарий с причиной доработки; сначала проверить задачу, результат и комментарии.
- `delete_bitrix_task` - удалить одну точную задачу; только после показа пользователю деталей задачи и явного подтверждения.

Если в запросе есть номер задачи (например, `318241`), это самый быстрый и правильный путь: сразу `search_tasks(bitrix_task_id=318241)` — мгновенный поиск по индексу, возвращает одну задачу. Не передавать номер в `query` (он ищет только по тексту названия/описания) и не листать все задачи через `offset`. Полный текст одной задачи: `search_tasks(bitrix_task_id=318241, include_full_description=true)`. Описания задач по умолчанию приходят сокращёнными (превью 500 символов) для скорости.

Использовать для: сроков, статусов, ответственных, просрочек, факта постановки задачи, контроля исполнения. Если известен сотрудник, сначала получить его `bitrix_user_id` через `get_org_structure`, затем искать задачи по `responsible_bitrix_user_id`.

Для комментариев, возобновления и удаления сначала найди ровно одну задачу по номеру и проверь название/статус/ответственного/дедлайн. Не изменяй задачу по названию, поисковой фразе или неоднозначной ссылке. Если нужного инструмента нет в твоём персональном наборе, прямо скажи, что эта операция недоступна текущему агенту.

### Чаты и сообщения
- `list_chats` - найти активные чаты и их `dialog_id`.
- `search_messages` - поиск по сообщениям и OCR вложений за период.
- `get_chat_transcript` - получить полную переписку одного чата за период, включая OCR при `include_ocr=true`.
- `get_chat_ocr_status` - проверить, готовы ли OCR-тексты изображений/PDF.
- `process_chat_ocr` - запустить OCR для вложений.
- `get_chat_daily_report` - прочитать сохраненный ежедневный отчет по чату.
- `save_chat_daily_report` - сохранить ежедневный отчет по чату.

Быстрый паттерн для вопроса по чату:
1. `list_chats(date_from,date_to,query)`.
2. `get_chat_transcript(dialog_id,date_from,date_to,include_ocr=true)`.
3. Если нужны вложения: `get_chat_ocr_status`, затем `process_chat_ocr` при необходимости.
4. При аналитике задач сверить с `search_tasks`.

### Zoom-созвоны
- `list_zoom_calls` - список Zoom-записей/созвонов за период.
- `get_zoom_call_transcript` - участники и транскрипт одного созвона.
- `search_zoom_transcripts` - поиск по сегментам Zoom-транскриптов.
- `save_zoom_call_report` - сохранить standalone Zoom-отчет.
- `delete_zoom_call_report` - удалить только AI-отчет по Zoom, не удаляя созвон и транскрипт.

Быстрый паттерн:
1. `list_zoom_calls(date_from,date_to)`.
2. Если знаешь тему/слово: `search_zoom_transcripts(query,date_from,date_to)`.
3. Для конкретного созвона: `get_zoom_call_transcript(call_id,include_full_text=true)`.
4. Для отчета: сначала `get_report_contract(category_key = "zoom_processing")`, затем сохранить через `save_zoom_call_report`.

### Связка чат + Zoom
Для ежедневного отчета по чату не передавать сырые Zoom-транскрипты напрямую в анализ чата. Если Zoom-отчета нет, сначала сформировать standalone Zoom-отчет, затем брать из него только то, что относится к конкретному чату: задачи участников, решения, блокеры, контрольные точки.

## Как искать быстро
- Начинай с узкого запроса: дата, чат, человек, проект, ключевое слово.
- Для периода сначала используй `get_period_index`, чтобы понять, где есть данные.
- Для людей сначала `get_org_structure`, потом задачи/чаты/Zoom.
- Для регламентов сначала `search_company_knowledge`, потом `get_company_file`.
- Для спорного вывода проверяй минимум два источника: например, задача Bitrix + чат; Zoom + задача; документ `О компании` + чат.
- Если поиск ничего не нашел, попробуй синонимы, фамилию/имя, проектное название и период шире.

## Источники в ответах
Всегда указывай, откуда факт:
- документ `О компании`: название/путь/URL;
- Bitrix: задача, ответственный, срок, статус;
- чат: `dialog_id`, дата, сообщение/OCR;
- Zoom: `call_id`, дата, таймкод/сегмент;
- оргструктура: ФИО, роль, отдел, руководитель.

Если источник не найден или данные противоречат друг другу, не сглаживай. Прямо напиши, что именно не найдено или конфликтует.
