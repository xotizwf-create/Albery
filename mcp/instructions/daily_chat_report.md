# Ежедневный отчет по чату: порядок работы ИИ

Эта инструкция не заменяет активный промпт `chat_analysis`. Она фиксирует порядок источников и правило работы с Zoom.

## Главное правило

Перед формированием ежедневного отчета по чату:

1. Вызвать `get_ai_prompts` с `category_key = "chat_analysis"`.
2. Использовать активный `prompt_text` как строгий контракт анализа и JSON.
3. Не формировать отчет по чату только по этой инструкции.

## Данные для отчета

Для отчета по конкретному чату за `report_date` нужны:

1. Чат и `dialog_id`.
2. Дневная переписка с OCR-текстом вложений.
3. Хвосты и отчет предыдущего календарного дня.
4. Для понедельника: недельный отчет предыдущей недели по этому чату и его открытые пункты.
5. Активные цели/контекст, если есть.
6. Zoom-отчеты текущего дня, а не сырые Zoom-транскрипты.

## Правило переноса хвостов

В контекст `Хвосты с прошлого дня` должны попасть все незакрытые пункты из предыдущего дневного отчета:

- `previous_day_tasks`;
- `next_steps`;
- `risks`.

Это контрольный список, а не только список строгих задач. Нельзя удалять пункт только потому, что у него нет срока, он был риском, был схлопнут с похожим пунктом или не подходит под `commitments`.

Для каждого перенесенного хвоста обязательно сохраняй счетчик `days_open`:

- если хвост появился впервые из предыдущего дня и счетчика нет, поставь `days_open = 1`;
- если в предыдущем отчете у хвоста было `days_open = N`, в новом отчете без движения должно быть `days_open = N + 1`;
- в текстовом отчете хвост со статусом `no_info` пиши не как `[тишина]`, а как `[тишина 1 день]`, `[тишина 2 дня]`, `[тишина 3 дня]` и так далее;
- не обнуляй `days_open`, если хвост перешел в новый день без прямого закрытия.

Для каждого хвоста нужно проверить текущий день:

- если есть явное закрытие - вернуть в `previous_day_tasks` со статусом `done`;
- если есть движение без закрытия - `in_progress`;
- если есть перенос - `postponed`;
- если есть блокер - `blocked`;
- если нет явного закрытия/движения - вернуть в `previous_day_tasks` со статусом `no_info`.

Риски предыдущего дня, по которым нет закрытия в текущем дне, тоже остаются в `previous_day_tasks` с `item_type = "risk"` и при необходимости дублируются в разделе `risks`.

## Правило понедельника

Если `report_date` - понедельник, перед ежедневным отчетом нужно использовать недельный отчет предыдущей недели по этому же чату.

Из недельного отчета обязательно перенести в контекст понедельника:

- `next_week_action_plan`;
- `hanging_tasks_by_owner`;
- `not_done`;
- `work_in_progress`;
- `hanging_goals`.

Эти пункты анализируются по тем же правилам, что и обычные хвосты предыдущего дня:

- проверить каждый пункт по переписке и Zoom-отчетам понедельника;
- если есть явное закрытие - вернуть в `previous_day_tasks` со статусом `done`;
- если есть движение без закрытия - `in_progress`;
- если есть перенос - `postponed` и новая дата, если она указана;
- если реакции нет - `no_info`;
- если есть блокер - `blocked`;
- не удалять недельный пункт без явного закрытия или отмены.

Если по недельному пункту в понедельник нет реакции, он должен остаться в `previous_day_tasks`, а при необходимости попасть в `risks` или `next_steps` с тем же владельцем и сроком.

## Формат пустого дня с хвостами

Если за дату нет сообщений/OCR/Zoom, но есть хвосты из предыдущего дня или недельного отчета, нельзя сохранять короткую заглушку вида "ничего нет".

Нужно сформировать полноценный ежедневный отчет:

1. `Короткий вывод` - новых данных нет, но хвосты остаются без движения.
2. `Источник` - чат, dialog_id, дата, сообщения/OCR/Zoom, предыдущий отчет и недельный контекст.
3. `1. Хвосты с прошлого дня` - все незакрытые хвосты со статусом `no_info`.
4. `2. Найденные цели` - новые цели или активные цели из предыдущих отчетов.
5. `3. Найденные задачи` - "новых задач нет" плюс задачи из предыдущего отчета на контроле.
6. `5. План выполнения задач/целей` - план по владельцам и срокам из незакрытых хвостов.
7. `6. Риски` - риски, которые сохраняются из-за отсутствия движения.
8. `7. Что контролировать на следующем отчете` - все контрольные пункты по владельцам и срокам.

## Правило Zoom

Для ежедневного отчета по чату сырая Zoom-транскрибация не должна быть основным источником анализа.

Алгоритм:

1. Найти Zoom-созвоны за `report_date` через `list_zoom_calls`.
2. Для каждого релевантного созвона, а если созвон за день один - для него обязательно, проверить наличие standalone Zoom-отчета в `analytical_note`.
3. Если Zoom-отчета нет, сначала сформировать standalone Zoom-отчет по инструкции `Формирование отчетов / Зум-созвоны` и сохранить его через `save_zoom_call_report`.
4. В `chat_analysis` передавать только метаданные созвона и готовый `analytical_note`.
5. Не передавать в `chat_analysis` полный `transcript_text` Zoom.

Из Zoom-отчета в отчет по чату брать только то, что относится к конкретному чату:

- задачи участников этого чата;
- выполнение, переносы и блокеры задач участников;
- решения по темам чата;
- риски по людям/процессам чата;
- контрольные точки, которые нужно проверить в следующем отчете по чату.

Не переносить в отчет по чату весь управленческий Zoom-отчет, общие темы компании и задачи людей, не относящиеся к этому чату.

## Сохранение

Итог сохранять через `save_chat_daily_report`.

В `raw_input` фиксировать:

- `zoom_call_id`;
- был ли Zoom-отчет создан перед отчетом по чату;
- какие `analytical_note` были переданы в `chat_analysis`;
- что сырая Zoom-транскрибация в `chat_analysis` не передавалась.
