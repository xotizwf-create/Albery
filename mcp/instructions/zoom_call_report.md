# Зум-созвон: порядок генерации отчета

Перед генерацией standalone-отчета по Zoom:

1. Вызвать `get_report_contract` с `category_key = "zoom_processing"` и использовать активный промпт как строгий контракт.
2. Получить созвон через `list_zoom_calls` и полный транскрипт через `get_zoom_call_transcript`.
   Если `total_segments` больше возвращенного `limit`, читать транскрипт постранично через `offset`
   до тех пор, пока не будут получены все сегменты. Лимит одного вызова не является блокировкой
   доступа и не должен описываться в отчете как недоступность полного транскрипта.
3. Обязательно вызвать `get_org_structure(include_inactive=false)`.
4. Передать в анализ:
   - фактических участников из Zoom API;
   - сегменты транскрипта;
   - оргструктуру с ФИО, ролями, отделами, руководителями и `bitrix_user_id`.

Отчет должен начинаться с:

- `Источник`;
- `Участники и упомянутые люди`.

В `Участники и упомянутые люди` разделять:

- `Участники созвона` - кто фактически был в Zoom по API/спикерам;
- `Упоминались в обсуждении` - люди, которых называли в разговоре.

Каждого человека нужно соотнести с оргструктурой. Если совпадение не найдено или неоднозначно, это должно быть явно написано.

В человекочитаемом `report_text` блок людей должен быть компактным. Не выводить служебные поля `raw_name`, `org_match`, `matched`, `ambiguous`, `not_found`, `bitrix_user_id`. Эти поля допустимы только в JSON. Видимый формат: `Евгений Палей - генеральный директор`, `Елена - требуется уточнение`, `Виктория - не найдена в оргструктуре`.

## Операционные задачи

Раздел `4. Операционные задачи` обязателен в каждом Zoom-отчете и в `report_text`, и в JSON.

В `report_text` каждая задача должна быть отдельной строкой строго в формате:

`N. Ответственный: <полное ФИО из оргструктуры или точное имя из разговора>. Задача: <короткое действие без технических слов>. Срок: <ДД.ММ.ГГГГ или срок не указан>. Критерий результата: <проверяемый результат>. Статус: planned. Источник: <таймкод или фрагмент>.`

Правила ответственных:

- Сначала сопоставь каждого ответственного с `org_context.users` / оргструктурой.
- Если человек найден в оргструктуре, используй полное ФИО из оргструктуры и его `bitrix_user_id`.
- Короткие имена и инициалы (`Настя`, `Артур С.`, `Дима`) можно использовать только как входной сигнал для сопоставления, но в JSON надо вернуть полное ФИО и `bitrix_user_id`, если совпадение однозначное.
- Нельзя назначать задачу роли без пользователя: `координатор`, `ответственный`, `команда`, `админ блок`, `юристы`, `фабрика`.
- Если ответственный не найден или совпадение неоднозначно, оставь имя из разговора, поставь `bitrix_user_id = null`, а в `result_criteria` или `source` кратко укажи, что требуется уточнение владельца.
- Не теряй задачи только потому, что ответственный не был фактическим участником Zoom. Если в разговоре назначили задачу Насте, Артуру, Наталье или Оксане, задача должна попасть в раздел и JSON.

Правила текста задач:

- Не использовать технические поля внутри текста задачи: `Ответственный`, `Задача`, `Срок`, `Статус`, `Источник` должны быть только структурными полями раздела 4 и JSON.
- `task_text` должен быть человеческим поручением без слов вроде `рекомендация`, `выделенная задача`, `операционная задача`, `система`, `ИИ`.
- `result_criteria` должен отвечать на вопрос, как понять, что задача выполнена.
- Если срок не прозвучал, писать `deadline_text = "срок не указан"`.
- Для задач, которые должны быть сделаны сегодня, указывать дату созвона в формате `ДД.ММ.ГГГГ`.

В JSON обязательно вернуть массив `operational_tasks` в том же порядке и с теми же задачами, что в разделе 4:

```json
{
  "operational_tasks": [
    {
      "number": 1,
      "assignee_name": "Полное ФИО из оргструктуры",
      "bitrix_user_id": 123,
      "task_text": "Проверить акции и отправить выводы на согласование",
      "deadline_text": "25.05.2026",
      "result_criteria": "Наталье отправлены выводы/скриншоты по акциям и скидкам на согласование",
      "status": "planned",
      "source": "00:05:15-00:05:34"
    }
  ]
}
```

Если `bitrix_user_id` не найден, вернуть `null`, но задачу не удалять. Каждая задача должна иметь `assignee_name`, `task_text`, `deadline_text`, `result_criteria`, `status`, `source`.

Раздел `Поведенческие факторы` обязателен. Его нельзя удалять из Zoom-отчета. Если значимых поведенческих сигналов нет, нужно прямо написать, что значимых сигналов не выявлено.
