# Ежедневный отчет по компании

Эта инструкция относится к разделу `Формирование отчетов / Ежедневный отчет по компании`.

## Главный принцип

Ежедневный отчет по компании формируется не из сырых данных, а из готовых отчетов-источников:

- дневной отчет каждого активного чата, где за дату есть сообщения;
- аналитический отчет каждого Zoom-созвона за дату;
- предыдущий ежедневный отчет для собственника;
- Bitrix как источник статусов, сроков и ответственных;
- оргструктура как источник реальных адресатов.

Общей сводки по всем чатам как обязательного источника нет. Нельзя заменять чтение отчетов конкретных чатов общей сводкой.

## Перед формированием

1. Открыть `Сводная аналитика / Настройка промтов / ежедневный общий отчет для собственника`.
2. Использовать активный ИИ-промт из этого раздела как основной контракт структуры, стиля и JSON.
3. Проверить, что за `report_date` сформированы daily-отчеты по всем активным чатам, где есть сообщения.
4. Проверить, что за `report_date` сформированы аналитические Zoom-отчеты по всем Zoom-созвонам.
5. Проверить, что сформирован предыдущий ежедневный отчет для собственника.

Если хотя бы один источник не готов, ежедневный отчет по компании не формировать. Сначала сформировать недостающий источник по его инструкции:

- для чата: инструкция ежедневного отчета по чату;
- для Zoom: инструкция отчета по Zoom-созвону;
- для предыдущего owner-отчета: эта же цепочка за предыдущую доступную дату.

## Что читать

- По чатам читать каждый `chat_daily_report` отдельно только для чатов с сообщениями за дату: полный текст отчета, хвосты, вопросы без ответа, решения, задачи, результаты, OCR-контекст.
- Активные чаты без сообщений за дату игнорировать: не создавать для них `no_data`, не считать их недостающим источником и не блокировать owner-отчет.
- По Zoom читать только готовый аналитический отчет (`analytical_note`). Сырую Zoom-транскрибацию в ежедневном отчете по компании не использовать.
- По предыдущему owner-отчету читать полный текст, особенно риски, рекомендации, пункт 6 и список проверок на завтра.
- По Bitrix проверять статусы, сроки, ответственных, просрочки и задачи без срока.

## Управленческая логика

- Сравнить текущий день с предыдущим owner-отчетом.
- Если адресная рекомендация из предыдущего owner-отчета не имеет явного подтверждения выполнения в текущих отчетах по чатам, Zoom или Bitrix, повторить ее в пункте 6.
- Не заменять старый незакрытый вопрос новой темой по тому же человеку.
- Новые устные договоренности из Zoom и чатов фиксировать как задачи, которые нужно занести в Bitrix.
- Отчет должен отвечать на вопросы: что сдвинулось, что зависло, где риск, кому какой вопрос задать.

## Запрещено

- Формировать отчет по одному чату, если активных чатов за дату несколько.
- Использовать общую сводку по всем чатам вместо отчетов конкретных чатов.
- Использовать сырую Zoom-транскрибацию.
- Писать техническую сводку: "найдено N чатов", "пайплайн", "коннектор", `get_period_index`, `search_tasks`.
- Строить отчет вокруг номеров Bitrix, Zoom call_id или dialog_id.
- Адресовать рекомендации ролям без пользователя: "Координатор", "ответственный", "административный блок", "юридический контур".

## Формат

Формат и JSON-схему брать из активного промта `ежедневный общий отчет для собственника`.

Обязательные смысловые блоки:

1. Главный вывод.
2. Динамика.
3. Риски.
4. Рекомендации.
5. Полный текст отчета с разделом `6. Адресные вопросы и рекомендации для Bitrix`.

Адресные рекомендации должны быть готовыми сообщениями реальным людям из оргструктуры Bitrix.
