# Агент воронки ИУ — как всё устроено от «привет» до подписания

Полное описание агента, который ведёт клиентов по воронке «Партнёрская программа WB —
индивидуальные условия» в Telegram от лица аккаунта компании. Бизнес-процесс, техническая
цепочка, все защитные механизмы. Написано 23.07.2026 по требованию владельца; каждый механизм
здесь существует в коде и закрыт тестами.

## Общая картина

Клиент пишет в личку менеджерского аккаунта Telegram. Отвечает не человек, а агент — от лица
аккаунта, без слова «бот». Агент сверяет анкету, отправляет условия дословно, собирает
реквизиты, формирует договор по шаблону владельца, отправляет PDF, ставит задачи сотрудникам
и сообщает клиенту о их выполнении, двигая сделку по стадиям CRM. Владелец видит всю переписку
в кабинете и весь процесс — в Битриксе.

```
Клиент в Telegram
   │  business_message (Telegram Business)
   ▼
albery-tg (tg_agent.py, systemd)            ←— цикл getUpdates, пул потоков
   │  буфер сообщений + пауза-добор 8 с      ←— пачка = один ход
   │  белый список: лид воронки? незнакомец?
   ▼
hermes CLI (subprocess, «мозг»)              ←— роль из карточки агента + STYLE_RULES
   │  + ТЕКУЩИЙ ШАГ ВОРОНКИ из сделки        ←— funnel_step_block, в каждом ходе
   │  + история диалога (12 записей)
   ▼
MCP приложения (mcp/context_server.py, albery.service, HTTP :5002)
   │  send_terms / send_contract / update_crm_deal / create_bitrix_task /
   │  notify_client_when_task_done / search_company_knowledge / …
   ▼
Bitrix24 (CRM, задачи) · Google Drive (база знаний) · PostgreSQL (журнал, ожидания)
```

Два процесса — это важно: **albery-tg** (Telegram-цикл) и **albery.service** (Flask + MCP).
Импортировать `mcp.context_server` или `bitrix` в процесс tg-агента нельзя — их импорт
запускает живые планировщики или циклится. Всё общение tg-агента с инструментами — по HTTP
(`mcp_call`) или голым REST (`_task_status`); общая память двух процессов — PostgreSQL.

## Бизнес-процесс по шагам (стадии воронки, category 16)

Источник истины о том, что делать сейчас, — **сделка в CRM**, не память модели. Перед каждым
ходом `funnel_step_block(deal_id)` читает сделку и кладёт в промпт текущий шаг: этап, чего
ждём от клиента, что сделать. Клиент может задать сколько угодно вопросов между шагами —
агент отвечает и возвращается к шагу.

| Стадия | Шаг | Что происходит |
|---|---|---|
| `C16:NEW`, `C16:CONTACTED` | Сверка анкеты | Агент показывает данные анкеты (магазин, категория, обороты), просит подтвердить. Подтвердил → стадия `C16:S84294149` + сразу `send_terms`. |
| `C16:S84294149` (условия не отправлены) | Отправка условий | `send_terms` шлёт текст **дословно** из документа базы знаний + «Есть вопросы по условиям?». Своими словами пересказывать запрещено. |
| `C16:S84294149` (условия отправлены) | Вопросы по условиям | Ответы только по базе знаний (`search_company_knowledge`). Вопросы кончились → просим реквизиты. |
| `C16:S84294149` (реквизиты пришли) | Договор | Реквизиты → поле сделки → `send_contract`: PDF по шаблону владельца уходит клиенту на согласование. |
| `C16:NDA` (способ не выбран) | Способ подписания | ЭДО или бумага. Выбрал → поле сделки + задача сотруднику «Направить договор на подписание» + `notify_client_when_task_done`. |
| `C16:NDA` (способ выбран) | На подписании | Сотрудник закрыл задачу → сторож сам сообщает клиенту (≤20 с). Клиент подписал → `C16:UC_SGZRVS`. |
| `C16:UC_SGZRVS` | Счёт | Задача бухгалтеру + ожидание закрытия → `C16:PREPAYMENT_INVOIC`. |
| `C16:PREPAYMENT_INVOIC` | Ожидание оплаты | Стадия двигается ТОЛЬКО по подтверждению бухгалтера — «я оплатил» от клиента не факт оплаты. |
| `C16:EXECUTING` | Подключение | Инструкции клиенту, техшаги → `C16:CONNECTED`. |

## Кто получает ответ: белый список

Аккаунт живой — туда пишут и поставщики, и знакомые. Автоответ получает только:

1. **Лид воронки** — его @username есть в сделке (поле `UF_CRM_1784296997`), сверка через
   `list_crm_lead_contacts` (кэш 5 минут). Точки/подчёркивания в username нормализуются
   (`griaznov.d` = `griaznov_d`). **При промахе кэш перечитывается немедленно** (не чаще раза
   в минуту): клиент заполнил анкету — сделка появилась сию секунду, ждать 5 минут нельзя.
2. **Незнакомец** — если включено `TG_LEAD_INVITE`: живой разговор по базе знаний, в первый
   ответ хвостом — ссылка на анкету (повторно — не раньше кулдауна). Заполнил анкету → сделка
   → дальше как лид.
3. **CRM недоступна** → агент молчит: лида от незнакомца не отличить, лучше пауза, чем
   неправильное сообщение поставщику.

Исходящие самого владельца, боты и группы игнорируются всегда (иначе агент отвечал бы сам себе).

## Человечное поведение (владелец, 23–24.07.2026)

- **Пачка сообщений = один ответ.** Сообщения человека копятся в буфере; окно паузы-добора
  (`TG_REPLY_DEBOUNCE_S`, 15 с) **скользит от последнего сообщения**: клиент пишет 7 сообщений
  с паузами меньше окна — агент молчит, пока тот не выговорится, и отвечает на всё одним
  сообщением. Ходы одного человека строго последовательны (`dialog_lock`), разных людей —
  параллельны (лимит `_HERMES_PARALLEL`, бокс 2 ГБ).
- **Клиент дописал, пока агент думал → ход перезапускается.** Прежний ответ выбрасывается
  НЕотправленным, пачка расширяется, шаг воронки перечитывается, мозг думает заново над всеми
  сообщениями. Лимит перезапусков `TG_RETHINK_MAX` (3) — остаток заберёт следующий ход.
- **Адекватность данных.** Реквизиты проверяет математика, а не глаз модели: контрольные
  суммы ИНН (10/12 цифр) и ОГРН (13/15), формат КПП/БИК/счетов, форма собственности в
  названии (`contract.validate_requisites`). Кривые реквизиты → договор НЕ собирается,
  агент называет клиенту каждую проблему по-русски. Мусор, не похожий на реквизиты, парсер
  не извлекает вовсе → агент просит карточку компании. Промпт запрещает модели «додумывать»
  («похоже, это ИНН?» — реальный случай 24.07.2026, диалог 6514126096).
- **Живой тон** — `STYLE_RULES` в каждом промпте: не начинать сообщения с имени клиента, не
  переспрашивать отвеченное, коротко и тепло, один встречный вопрос за раз.
- **Реакция 👀** на сообщение лида — клиент видит, что его прочитали, ещё до ответа.
- **История** — последние 12 записей журнала в каждом промпте; служебные записи об эскалации
  в историю не попадают (клиент их не видел).

## Условия — дословно (`send_terms`)

Текст условий живёт в Google-документе базы знаний «Условия ИУ — текст для клиента»
(`TERMS_DOC_NAME`). Клиенту уходит только то, что ниже строки-маркера `--- ТЕКСТ КЛИЕНТУ ---`,
дословно, без пересказа моделью. Защиты: нет маркера → не отправляем ничего; в тексте остались
пометки `[ЗАПОЛНИТЬ]` → не отправляем и говорим владельцу. Отправка отмечается в сделке.
Владелец правит документ на Drive — агент шлёт новое без деплоя.

## Договор — по шаблону владельца (`send_contract`)

1. Реквизиты клиента парсятся из свободного текста/карточки компании (`parse_requisites`);
   не хватает — агент спрашивает именно недостающее (без «пришлите всё заново»).
2. Текст берётся из Google-документа «Шаблон договора ИУ»; плейсхолдеры `{ИНН}`, `{НАЗВАНИЕ}`…
   заполняются данными клиента, `{ИСПОЛНИТЕЛЬ — …}` — реквизитами Албери (env `CONTRACT_*`).
   Длинные плейсхолдеры подставляются первыми (иначе `{ИСПОЛНИТЕЛЬ — ИНН}` частично затёрся бы
   `{ИНН}`); «ООО «ООО …»» схлопывается; строки с незаполненными плейсхолдерами убираются.
3. PDF собирает reportlab (LibreOffice на бокс 2 ГБ не встал): таблицы шаблона — настоящими
   таблицами, заголовки по центру, «Приложение № 1» с новой страницы, шрифт Liberation Serif.
4. Номер — `next_contract_number` («ДД.ММ.ГГГГ», «ДД.ММ.ГГГГ-1», …) по уже выданным номерам.
5. PDF уходит клиенту, кладётся в сделку (поле «Договор — файл»), пишутся номер и дата,
   стадия — `C16:NDA`. Незаполненные места шаблона возвращаются агенту — он говорит владельцу,
   а не молчит.

## Сторож анкеты: агент замечает заполненную анкету сам

Незнакомец получил ссылку на анкету (`state.invited`) → заполнил → в CRM появилась сделка.
Ждать, пока человек напишет «готово», не нужно: сторож анкеты (`_check_new_forms`, тем же
потоком, что сторож задач, раз в ~минуту) сам замечает новую сделку приглашённого и отправляет
сверку. Одна сделка обрабатывается один раз (`state.form_surveyed[uid] = deal_id`); повторная
анкета создаёт новую сделку — сторож срабатывает снова. Защиты: агент уже говорил по этой
сделке (в журнале есть исходящее с её `deal_id`) — молчит; сделка уже дальше сверки — не
лезет; с человеком идёт ход (`dialog_lock` занят) — пропускает тик.

**Сверка — из живых полей воронки.** Текст «Вижу анкету: • \<поле\> — \<значение\> … Всё
верно?» собирает код (`anketa_block`), а не модель: состав полей — env `CRM_FORM_FIELDS`,
названия — из `crm.deal.fields` живьём (кэш 10 мин). Переименовали поле в CRM — сообщение
поменялось само, без деплоя. Числа по-человечески: 5000000 → «5 млн». Пустые поля не
показываются. Тот же блок вставлен в шаг воронки «Сверка анкеты» — сторож и обычный ход шлют
один и тот же текст дословно.

## «Сотрудник закрыл задачу → клиент узнаёт сам»

Механизм из двух частей (оба конца обязательны):

1. **Регистрация**: создав задачу сотруднику, агент вызывает `notify_client_when_task_done`
   (задача, telegram клиента, текст, опционально `next_stage`) → строка в `funnel_task_watch`.
2. **Сторож** (`_task_watch_loop`, поток в albery-tg, каждые 20 с — `TG_TASK_WATCH_INTERVAL_S`):
   задача закрыта (статус 4/5 через голый REST `tasks.task.get`) → сообщение клиенту → отметка
   `notified_at` → стадия `next_stage` через `mcp_call`. Защиты: два ожидания одного смысла
   (сделка + вид) → второе снимается, клиент не получает дубль; новая регистрация заменяет
   старую; задача удалена из портала → ожидание снимается; журнал недоступен → ничего не
   гасим и не шлём лишнего; `telegram_id` меньше 100000 отклоняется ещё на регистрации (модель
   однажды передала Bitrix-id сотрудника — задача 2018), а недоставимый адрес
   (`PEER_ID_INVALID`) снимает ожидание вместо вечных повторов.

## Дубли исключены по построению

`send_terms`/`send_contract` сами пишут клиенту (из процесса MCP). Чтобы модель не отправила
следом «условия отправили вам сюда» вторым сообщением: перед ходом снимается отметка журнала
(`_dialog_out_watermark`), после хода — если в диалоге появились новые исходящие
(`_out_messages_after`), финальная реплика модели **не отправляется**, а входящие клиента всё
равно журналируются. Плюс дедуп ожиданий сторожа (выше). Итого одно событие = одно сообщение.

## Эскалация к людям

- **`НУЖЕН_ЧЕЛОВЕК: <суть>`** — в базе знаний ответа нет. Клиенту — тишина (обещание «уточню
  и вернусь» хуже паузы), вопрос уходит карточкой в группу Битрикса «Работа с ИУ» с куском
  переписки и telegram id клиента. Сотрудник отвечает в группе «<Агент>, ответь, что …» —
  агент передаёт клиенту.
- **`ТАКЖЕ_СПРОСИ_ЛЮДЕЙ: <вопрос>`** — ответить есть что, но не хватает конкретики: клиент
  получает ответ по существу, вопрос параллельно уходит людям.
- Группа недоступна → запасной канал: `TG_ESCALATION_CHAT_ID` или владелец напрямую.

## Журнал и наблюдаемость

- **`telegram_bot_messages`** (PostgreSQL) — вся переписка с участием агента (`kind=lead_chat`):
  каждое сообщение пачки отдельной записью, `meta` несёт deal_id/terms/contract/task_closed/
  escalated. Личные чаты аккаунта без участия агента в журнал не пишутся (решение владельца).
  Кабинет показывает переписку отсюда.
- **`funnel_task_watch`** — ожидания закрытия задач (см. сторожа).
- Логи: `journalctl -u albery-tg` (ходы, сторож, пачки), `journalctl -u albery.service` (MCP).

## Правила изменения этого агента

1. Первопричина только по фактам: журнал (`telegram_bot_messages`), лог службы, сделка в CRM.
2. Красный тест, воспроизводящий жалобу реальными данными → фикс → зелёный.
3. Полный прогон + `scripts/predeploy_check.py` **не на проде** — красные ворота = нет деплоя.
4. После деплоя — проверка на проде на тех же реальных данных, что в жалобе.
5. Каждое изменение — закрытая задача в Битриксе + строка в changelog мозга
   (`projects/albery/change-tracking.md` в hermes-brain).
6. Тесты агента: `tests/unit/test_tg_*.py`, `tests/unit/test_contract*.py`,
   `tests/unit/test_terms.py`, `tests/unit/test_funnel_step.py`.
