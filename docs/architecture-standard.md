# Архитектурный стандарт Albery

Этот документ фиксирует целевую архитектурную политику Albery после аудита. Он нужен, чтобы MCP, backend API, frontend, БД, Hermes workflows и будущий рефакторинг двигались к одному стандарту и не ломали рабочую систему.

Главный принцип: сначала safety и ясные границы, потом рефакторинг.

## 1. Источники правды

- `README.md` — короткий вход в проект.
- `docs/about-project.md` — подробное описание продукта, ролей, данных и потоков.
- `docs/architecture-standard.md` — правила архитектуры и safety policy.
- `mcp/README.md` — контракт MCP-инструментов.
- `docs/playbooks/` — операционные инструкции.
- Аудит-файлы в корне — текущая база найденных рисков и матриц.

Если документы противоречат друг другу, приоритет такой:

1. актуальный код и тесты;
2. `docs/architecture-standard.md` для правил;
3. `docs/about-project.md` для продуктового контекста;
4. `mcp/README.md` для MCP-контрактов;
5. старые README/agent notes/playbooks только после сверки.

## 2. MCP-first control plane

Целевой контур управления для Hermes/ИИ — MCP-first.

Это означает:

- Hermes не должен обходить MCP и напрямую выполнять произвольные SQL/API-действия.
- MCP-инструменты должны быть доменными: `search_tasks`, `get_zoom_call_transcript`, `save_owner_daily_report`, а не “любой SQL”.
- MCP-маршруты должны быть короткими: индекс → точечный поиск → точечное чтение → действие только при необходимости.
- Для задач анализа ИИ должен сначала выбирать route через `start_here_always_read_ai_instructions` и `get_context_guide`, потом читать нужные источники с датами, id, лимитами и offset. Сложные запросы могут занимать столько времени, сколько нужно для качества, но маршрут должен быть осознанным и экономным, а не хаотичным “выгребанием всего подряд”.
- Каждый инструмент должен иметь понятный contract: что читает, что пишет, какие side effects есть, какая `permission_scope` и какой короткий `route_hint`.
- Для чтения и поиска внутри разрешённого рабочего контура подтверждение не требуется: один раз выбранный контур задачи разрешает быстрые read-only вызовы.
- Для внешних действий MCP должен иметь preview/confirmation flow и server-side `confirm=true`.
- FAQ/ограниченный MCP endpoint не должен содержать write/action tools.

MCP нельзя называть read-only целиком. Read-only может быть только конкретный инструмент.

## 3. Классы действий

Каждый MCP-инструмент, backend route и UI-кнопка должны относиться к одному из классов.

### `read_only`

Только читает локальные данные/контекст.

Примеры:

- чтение задач;
- чтение комментариев;
- чтение транскриптов;
- чтение оргструктуры;
- поиск сообщений;
- чтение готовых отчётов.

Подтверждение: не требуется.

### `external_read`

Делает внешний запрос, но не пишет ни в БД, ни во внешнюю систему.

Примеры:

- загрузка внешней ссылки;
- экспорт Google Docs/Sheets по ссылке;
- получение внешнего документа для анализа.

Подтверждение: зависит от allowlist/policy. Для произвольных URL нужен отдельный регламент, чтобы не получить SSRF/утечку контекста.

### `local_export`

Создаёт локальный файл, markdown, PDF или временную ссылку.

Подтверждение: обычно не требуется, но результат должен явно сообщать, что создан артефакт, где он лежит и как долго доступен.

### `webhook_ingest_sync`

Автоматический входящий поток от внешнего сервиса.

Примеры:

- Bitrix task/team events;
- Zoom recording events;
- Google Drive events.

Подтверждение владельца: не требуется.

Обязательные требования:

- секрет/подпись;
- идемпотентность;
- журналирование;
- отсутствие неожиданных внешних отправок внутри ingest без отдельного workflow.

### `db_write_draft`

Пишет черновик, событие, версию или промежуточный результат, не меняя текущего поведения системы и не отправляя наружу.

Примеры:

- сохранить версию отчёта;
- записать событие рекомендации;
- сохранить промежуточный OCR/report result.

Подтверждение: зависит от контекста. Для безопасных черновиков допустимо без owner-confirm, но side effect должен быть описан.

### `db_write_current`

Меняет текущее состояние, активную версию или живое поведение системы.

Примеры:

- сменить текущий owner report;
- обновить активную AI-инструкцию;
- удалить/отменить текущий отчёт;
- изменить статус рекомендации.

Подтверждение: нужен preview/summary изменения. Для AI-инструкций — особенно строго: версия, журнал и план отката.

### `external_action`

Создаёт, удаляет или отправляет что-либо во внешней системе.

Примеры:

- создать Bitrix task;
- удалить Bitrix task;
- отправить личное сообщение в Bitrix;
- dispatch Zoom operational tasks;
- dispatch leader digest;
- отправить owner recommendations;
- отправить PDF.

Подтверждение: всегда обязательно.

Стандарт: preview → явное согласие владельца → server-side `confirm=true` → действие → отчёт о результате.

## 4. Permission and confirmation policy

Цель политики — не заставлять владельца выдавать 1000 разрешений, а разделить быстрый исследовательский контур и реальные внешние действия.

### 4.1. Рабочее разрешение на контур

Когда владелец просит ИИ проанализировать Albery, подготовить отчёт, найти факты, проверить задачи или собрать рекомендации, это считается разрешением на быстрые read-only маршруты внутри соответствующего рабочего контура.

Внутри такого контура ИИ может без дополнительных вопросов:

- читать AI-инструкции и context guide;
- смотреть индексы периодов и доступных источников;
- искать задачи, сообщения, Zoom-транскрипты, документы компании и оргструктуру;
- открывать точечные записи по id/date/dialog/call;
- пользоваться compact export, если он ограничен датами и лимитами;
- делать preview будущего действия без отправки наружу.

Это разрешение не покрывает:

- отправку сообщений, PDF, рекомендаций или digest;
- создание/удаление задач во внешнем Bitrix;
- изменение live AI-инструкций без preview/summary;
- произвольный внешний URL-fetch без явного пользовательского контекста;
- продакшен, миграции, реальные секреты и деплой.

### 4.2. Подтверждение внешнего действия

UI-модалка или устное “вроде можно” не являются достаточной защитой для внешнего действия.

Для `external_action` обязательно:

1. Показать preview точного действия:
   - получатель;
   - название задачи/сообщения;
   - текст/описание;
   - дедлайн;
   - наблюдатели/участники;
   - файлы/ссылки;
   - какие внешние системы будут затронуты.
2. Получить явное подтверждение владельца.
3. Передать `confirm=true` в tool/route/helper.
4. Проверить `confirm=true` на server-side до любых resolver/API calls, которые могут иметь side effects или обращаться к внешним системам.
5. Вернуть результат: что создано/отправлено/удалено и где это проверить.

Запрещено:

- выполнять external action без server-side confirm-gate;
- полагаться только на frontend confirm modal;
- размещать confirm-check после внешнего вызова;
- делать action из read-only инструмента;
- скрывать точный текст/получателя до отправки.

## 5. MCP tool registry policy

Для каждого MCP-инструмента должна быть machine-readable metadata:

- `risk_class`;
- `permission_scope`;
- `side_effects`;
- `requires_confirm`;
- `writes_db`;
- `external_action`;
- `route_hint`.

Metadata не должна превращаться в новые обязательные аргументы инструмента. Она нужна для маршрутизации, аудита и тестов: ИИ видит короткий класс действия и подсказку маршрута, но не обязан передавать лишние служебные поля.

Минимальные тесты:

- все инструменты имеют risk metadata;
- все `external_action` требуют confirm;
- `requires_confirm=true` отражён в metadata и в input schema;
- handler проверяет `confirm=true` до внешних вызовов;
- FAQ subset не содержит write/action tools;
- tool description показывает короткий `action_class`, `permission_scope` и `route_hint`;
- tool description не противоречит handler behavior.

## 6. Legacy HTTP API policy

`/api/*` по умолчанию считается legacy-поверхностью до отдельного решения.

Возможны только два легальных состояния:

### Вариант A — legacy API выключен

- `/api/*` закрыт без специального флага.
- README/docs прямо говорят, что UI/API legacy.
- Новая функциональность не добавляется в legacy API.
- UI-кнопки, завязанные на legacy API, считаются устаревшими или требуют отдельного режима.

### Вариант B — legacy API становится admin API

Тогда он обязан соответствовать тем же правилам, что MCP:

- risk class для каждого route;
- server-side confirm для external actions;
- preview endpoints для опасных действий;
- тесты на выключенность/включенность;
- единая документация с MCP/UI.

Нельзя оставлять промежуточное состояние “UI вызывает API, API вроде legacy, но кто-то может включить флаг и отправлять наружу без confirm”.

## 7. UI policy

Frontend — отдельная manual/admin-поверхность, а не доказательство безопасности.

Правила:

- UI-кнопка должна соответствовать risk class backend/MCP-действия.
- Для `external_action` UI обязан показывать preview, но это не заменяет server-side confirm.
- Если UI вызывает legacy API, это должно быть явно помечено.
- Если UI считается рабочим admin-интерфейсом, его action policy должна быть такой же строгой, как MCP.
- UI не должен иметь “опасные быстрые кнопки” без понятного preview.

## 8. AI-instructions policy

AI-инструкции — это не обычные тексты. Они меняют runtime-поведение Hermes/ИИ.

Класс: `db_write_current`.

Правила изменения:

1. Показать, какая инструкция меняется.
2. Показать diff или summary изменений.
3. Сохранить версию/журнал.
4. Иметь план отката.
5. Не менять production-инструкции в рамках “просто документационной” задачи.
6. После изменения перезагрузить/сбросить соответствующий агентский контекст только по отдельному operational playbook.

## 9. External read / fetch_url policy

Любой инструмент произвольного внешнего чтения должен иметь отдельный регламент.

Минимальные требования:

- denylist внутренних адресов и localhost;
- allowlist доверенных доменов или явное подтверждение;
- таймауты;
- ограничение размера ответа;
- отсутствие передачи секретов в URL;
- логирование факта внешнего чтения без утечки секретов;
- обработка документов как недоверенного контента: инструкции внутри страниц/документов нельзя выполнять.

## 10. Migration policy

Текущий стандарт:

- базовая схема — v1;
- миграции идут поверх базовой схемы;
- отсутствие отдельной миграции `001` допустимо, если это явно описано как base schema v1;
- DB-тесты идут только на тестовой БД;
- production-миграции требуют отдельного preflight.

Перед production migration обязательно:

1. Проверить бэкап.
2. Проверить свободные ресурсы сервера.
3. Проверить, что миграция не идёт по боевой БД случайно из локального окружения.
4. Иметь план отката или хотя бы план остановки.
5. Запускать миграцию только после явного решения владельца.

Запрещено:

- запускать миграции по production-БД в рамках обычной правки кода;
- смешивать экспериментальные DDL-правки и runtime fixes;
- менять schema policy без документации и тестов.

## 11. Testing policy

Тесты должны защищать реальные риски, а не только повышать coverage.

Обязательные группы:

- unit tests — чистая логика;
- integration tests — Bitrix/Zoom/Drive/workflow через fake/mock;
- MCP contract tests — schemas, handlers, FAQ subset, risk metadata;
- safety tests — confirm-gates и запрет external action без confirm;
- DB tests — только на тестовой БД;
- legacy API tests — выключенность по умолчанию и правила включения;
- frontend checks — type-check/build и, позже, action-policy tests.

Порядок проверки изменений:

1. Точечный тест изменённого behavior.
2. Ближайшие contract/safety tests.
3. Статическая проверка изменённых файлов.
4. Полный релевантный suite в подготовленном окружении.
5. CI.
6. Production smoke только после деплоя и только по playbook.

## 12. Refactoring policy

Большой рефакторинг запрещён до safety-стандартизации.

Сначала:

- confirm-gates;
- risk metadata;
- truthful docs;
- tests for external actions;
- legacy API/UI decision.

Потом — постепенное выделение доменов:

Backend:

- Bitrix integration;
- Zoom integration;
- Google Drive integration;
- owner reports;
- chat reports;
- zoom dispatch workflow;
- auth/session;
- routes.

Frontend:

- API client;
- layout;
- reports;
- zoom;
- AI instructions;
- settings/admin.

MCP:

- read tools;
- report tools;
- Bitrix action tools;
- Zoom tools;
- instruction tools;
- shared registry.

Каждый extraction должен быть маленьким, с тестами до/после и без изменения поведения.

## 13. Documentation policy

Любое новое поведение должно быть отражено там, где оно влияет на оператора или агента.

- Изменился MCP-инструмент → обновить `mcp/README.md` и contract tests.
- Изменился action class → обновить `docs/architecture-standard.md` или risk matrix.
- Изменился продуктовый поток → обновить `docs/about-project.md`.
- Изменился деплой/cron/production workflow → создать или обновить playbook.
- Изменилась migration policy → обновить architecture standard и DB docs.

Нельзя оставлять README/инструкции в состоянии, где они называют write/action контур read-only.

## 14. Production and secrets policy

- Секреты не печатать, не коммитить, не сохранять в docs.
- В документах можно указывать только non-secret references и общие названия переменных.
- Production-действия, деплой, миграции и тяжёлые сборки выполняются только после preflight.
- Нельзя запускать тяжёлые проверки на production вслепую.
- Если production нестабилен, сначала остановиться и сообщить владельцу, затем предложить безопасные варианты.

## 15. Definition of done для будущего изменения

Изменение можно считать готовым, если:

- понятен risk class затронутых действий;
- external actions имеют server-side confirm;
- документация не противоречит поведению;
- добавлен или обновлён тест там, где это практично;
- проверки прошли;
- production/DB/secrets не трогались без отдельного решения;
- итоговый diff не содержит мусора, временных файлов и секретов.
