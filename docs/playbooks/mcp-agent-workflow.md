# Playbook работы ИИ-агента через MCP

Этот playbook описывает, как ИИ-агент должен работать с Albery через MCP: быстро, системно, без хаотичного чтения всего подряд и без опасных действий без подтверждения.

Главная цель: агент должен понимать, какой маршрут выбрать под задачу, какие инструкции обязательно прочитать, где остановиться на preview и когда можно действовать без лишних вопросов.

## 1. Главный принцип

Сложный запрос может занимать столько времени, сколько нужно для качества, особенно если нужно проверить много источников, периодов, задач, созвонов или регламентов.

Но агент не должен тратить время хаотично. Правильный подход:

1. прочитать обязательный входной контракт;
2. выбрать intent и короткий маршрут;
3. сначала открыть индексы/готовность/списки;
4. читать детали только по датам, id, диалогам, созвонам и задачам, которые нужны для ответа;
5. использовать лимиты, offset и компактные экспорты осознанно;
6. не выполнять внешние действия без preview и явного подтверждения владельца.

Не существует искусственного правила “после N минут остановиться”. Есть другое правило: если задача сложная, агент работает глубже, но обязан объяснимо двигаться по маршруту, а не “выгребать всё подряд”.

## 2. Источники правды и приоритет

Если источники противоречат друг другу, использовать такой порядок:

1. актуальный код, тесты и текущие MCP tool schemas;
2. `docs/architecture-standard.md` — правила безопасности, классы действий, confirmation policy;
3. live AI instructions из раздела “Настройки → Инструкции для ИИ”;
4. активные report contracts/prompts, которые возвращают MCP-инструменты;
5. `mcp/README.md` — общий MCP-контракт и список инструментов;
6. локальные инструкции в `mcp/instructions/`;
7. playbooks в `docs/playbooks/`;
8. старые README/agent notes — только после сверки.

## 3. Обязательный старт любого анализа

Для любой задачи из классов “анализ компании”, “отчет”, “рекомендации”, “найти факт”, “разобрать Zoom”, “подготовить действие” агент начинает так:

1. Вызвать `start_here_always_read_ai_instructions`.
   - Это живой контракт из настроек Albery.
   - Нельзя заменять его памятью, старыми заметками или локальным README.
2. Вызвать `get_context_guide(intent=...)`.
   - Если intent понятен — передать конкретный intent.
   - Если intent непонятен — сначала получить общий guide или задать один короткий уточняющий вопрос.
3. Проверить risk metadata инструментов.
   - Смотреть `risk_class`, `permission_scope`, `side_effects`, `requires_confirm`, `writes_db`, `external_action`, `route_hint`.
   - Metadata нужна для маршрутизации и проверки безопасности; это не дополнительные аргументы, которые надо передавать в каждый инструмент.
4. Использовать индексы перед деталями.
   - Для периода: `get_period_index`, `get_report_readiness`.
   - Для чатов: `list_chats`, потом точечный `get_chat_transcript`.
   - Для Zoom: `list_zoom_calls`, потом точечный `get_zoom_call_transcript` или `search_zoom_transcripts`.
   - Для документов: `list_company_files`, `search_company_knowledge`, потом `get_company_file`.
   - Для задач: `search_tasks`, потом `get_task_comments` по конкретной задаче.

## 4. Обязательные инструкции по типам задач

### 4.1. Универсальные live-инструкции

Перед любым содержательным ответом по Albery обязательно читать live AI instructions через `start_here_always_read_ai_instructions`.

Эти инструкции могут быть изменены владельцем в интерфейсе, поэтому агент не должен полагаться только на документацию в репозитории.

### 4.2. Ежедневный отчет по компании

Перед owner daily report обязательно использовать инструкцию:

- `mcp/instructions/company_daily_report.md`;
- live-раздел “Формирование отчетов / Ежедневный отчет по компании”;
- активный промт из “Сводная аналитика / Настройка промтов / ежедневный общий отчет для собственника”.

Критичные правила:

1. Ежедневный отчет по компании формируется из готовых отчетов-источников, а не напрямую из сырого потока.
2. Сначала вызвать `get_report_readiness(date_from=report_date, date_to=report_date)`.
3. Если не готовы daily-отчеты активных чатов, аналитические Zoom-отчеты или предыдущий owner daily report — owner daily report не формировать.
4. Нельзя заменять отчеты конкретных чатов общей сводкой.
5. Сырые Zoom-транскрипты в owner daily report не использовать; использовать готовые аналитические Zoom-отчеты.
6. Адресные рекомендации доставляются задачами в Bitrix, а не личными сообщениями.
7. Для доставки рекомендаций нужен preview и подтвержденное external action.

### 4.3. Zoom-отчет

Перед standalone-отчетом по Zoom обязательно использовать инструкцию:

- `mcp/instructions/zoom_call_report.md`;
- `get_report_contract(category_key="zoom_processing")`.

Критичные правила:

1. Получить созвон через `list_zoom_calls` и полный транскрипт через `get_zoom_call_transcript`.
2. Если сегментов больше, чем вернул один вызов, дочитать постранично через `offset`; лимит одного вызова не считается недоступностью полного транскрипта.
3. Обязательно прочитать оргструктуру через `get_org_structure(include_inactive=false)`.
4. Участников и упомянутых людей сопоставлять с оргструктурой.
5. Раздел операционных задач обязателен.
6. Не назначать задачи ролям без конкретного человека.
7. Если человек не найден или совпадение неоднозначно — явно указать, что нужно уточнение, но не терять задачу.

### 4.4. Выгрузки и большие файлы

Перед задачами на выгрузку больших Markdown/отчетов/транскриптов использовать playbook “Передача файлов через МСП”.

Критичные правила:

1. Большой результат не возвращать инлайном.
2. Сохранить файл и вернуть короткий preview плюс ссылку/артефакт.
3. Для набора конкретных дат использовать прямой селектор дат, а не листать созвоны ради ручного сбора id.
4. Один пользовательский запрос на выгрузку должен приводить к одному осмысленному экспорту, если нет причины дробить.

### 4.5. Инструкции, которые меняют живое поведение ИИ

`upsert_ai_instruction` относится к изменению текущего поведения системы.

Перед таким действием обязательно:

1. показать, какой раздел инструкции меняется;
2. показать краткое содержание изменения;
3. объяснить, что это повлияет на будущие ответы агента;
4. получить явное подтверждение владельца или оператора;
5. сохранить версию/журнал, если workflow это поддерживает;
6. иметь план отката.

## 5. Маршруты по intent

### 5.1. Найти факт по задаче, человеку или процессу

1. `start_here_always_read_ai_instructions`.
2. `get_context_guide(intent="fact_lookup")` или ближайший доступный intent.
3. Если вопрос про задачу — `search_tasks`.
4. Если нужен контекст — `get_task_comments` по конкретной задаче.
5. Если вопрос про человека/роль — `get_org_structure`.
6. Если вопрос про регламент — `search_company_knowledge`, затем `get_company_file`.
7. Ответ давать с человеческими названиями задач, владельцами, сроками и источниками, а не только с id.

### 5.2. Проверить дату или период

1. `start_here_always_read_ai_instructions`.
2. `get_context_guide(intent="period_analysis")` или ближайший доступный intent.
3. `get_period_index(date_from, date_to)`.
4. `get_report_readiness(date_from, date_to)`, если задача связана с отчетами.
5. Читать только релевантные чаты, Zoom, отчеты и задачи.
6. Если период большой — двигаться партиями и фиксировать, что уже проверено.

### 5.3. Подготовить owner daily report

1. `start_here_always_read_ai_instructions`.
2. `get_context_guide(intent="owner_daily_report_creation")`.
3. Прочитать инструкцию daily report и активный report prompt.
4. `get_report_readiness(report_date, report_date)`.
5. Если есть недостающие источники — остановиться и назвать, что нужно сформировать сначала.
6. Прочитать готовые daily-отчеты активных чатов.
7. Прочитать готовые Zoom analytical notes.
8. Прочитать предыдущий owner daily context.
9. Проверить Bitrix задачи, статусы, сроки и ответственных.
10. Проверить релевантные регламенты.
11. Сформировать preview/черновик.
12. Сохранять или доставлять только по правилам класса действия.

### 5.4. Разобрать Zoom-созвон

1. `start_here_always_read_ai_instructions`.
2. `get_context_guide(intent="zoom_report_creation")` или ближайший доступный intent.
3. `get_report_contract(category_key="zoom_processing")`.
4. `list_zoom_calls` для выбора созвона.
5. `get_zoom_call_transcript` с постраничным дочитыванием, если нужно.
6. `get_org_structure(include_inactive=false)`.
7. Сформировать отчет по инструкции Zoom.
8. Если нужно создать задачи в Bitrix — сначала preview, потом подтверждение, потом `confirm=true`.

### 5.5. Подготовить рекомендации руководителю/сотрудникам

1. `start_here_always_read_ai_instructions`.
2. `get_context_guide(intent="recommendations")` или ближайший доступный intent.
3. Прочитать предыдущий owner context и последние релевантные отчеты.
4. Проверить задачи и обратную связь по прошлым рекомендациям.
5. Проверить регламенты и оргструктуру.
6. Сформировать рекомендации как черновик/preview.
7. Для отправки наружу или создания Bitrix-задач — только external action flow.

### 5.6. Подготовить внешнее действие

Примеры: создать задачу, удалить задачу, отправить digest, отправить рекомендации, отправить PDF.

Обязательный маршрут:

1. Собрать основание и источники.
2. Сформировать preview:
   - что будет создано/отправлено/удалено;
   - кому;
   - точный текст;
   - дедлайн;
   - наблюдатели/участники;
   - файлы/ссылки;
   - внешняя система.
3. Получить явное подтверждение владельца.
4. Вызвать действие с `confirm=true`.
5. Вернуть результат и способ проверки.

Запрещено выполнять external action только потому, что пользователь попросил “подготовь” или “собери”. Подготовить — значит сделать preview, а не отправить.

## 6. Когда можно не спрашивать подтверждение

Подтверждение не нужно для read-only действий внутри понятного рабочего контура:

- читать live AI instructions;
- читать context guide;
- смотреть индексы и готовность отчетов;
- искать задачи и комментарии;
- читать чаты, отчеты, Zoom-транскрипты, документы компании;
- сверять оргструктуру;
- готовить черновик или preview.

Это правило нужно, чтобы владелец не подтверждал “тысячу чтений” во время нормального анализа.

## 7. Когда подтверждение обязательно

Подтверждение обязательно для:

- создания/удаления задач во внешнем Bitrix;
- отправки сообщений, PDF, рекомендаций, digest;
- изменения live AI-инструкций;
- изменения текущей версии owner report или другого текущего состояния;
- произвольного внешнего URL-fetch, если ссылка/домен не следует явно из задачи;
- продакшена, деплоя, миграций, секретов и действий с реальной инфраструктурой.

Для external action всегда нужен полный flow:

`preview → явное подтверждение → confirm=true → действие → отчет о результате`.

## 8. Как не превращать сложный запрос в хаос

Если задача широкая:

1. Сначала определить intent и границы: дата, период, человек, отдел, процесс, источник.
2. Если границы не указаны и их нельзя вывести из запроса — задать один короткий уточняющий вопрос.
3. Если границы понятны — не спрашивать лишнее, а работать.
4. Сначала получить индекс/готовность/список.
5. Читать детали только там, где индекс показал наличие данных или риск.
6. При большом объеме использовать pagination и фиксировать, какие страницы закрыты.
7. Не делать `list_*` только ради ручного сбора id, если есть прямой селектор дат/периода/источника.
8. Не возвращать огромный текст в чат, если нужен файл.
9. Если источник недоступен или не готов — честно сказать это и назвать следующий безопасный шаг.

## 9. Формат ответа владельцу

Ответ владельцу должен быть человеческим:

- без внутренних id как главной опоры;
- без технической сводки вызовов инструментов;
- с понятными выводами, рисками и следующими действиями;
- с указанием, что именно не было сделано, если это важно для безопасности.

Если действие только подготовлено, писать “подготовлено к подтверждению”, а не “сделано”.

Если внешнее действие выполнено, указать:

- что выполнено;
- где проверить;
- что не затрагивалось лишнего.

## 10. Definition of done для агентской работы

Работа считается завершенной, когда:

1. нужные инструкции были прочитаны;
2. выбранный маршрут соответствует intent;
3. источники проверены точечно, а не случайным сканированием;
4. все найденные ограничения явно отражены;
5. черновик/action preview отделен от реального external action;
6. внешние действия выполнены только после подтверждения;
7. результат можно проверить по источникам или созданному артефакту;
8. документация/инструкция обновлена, если в процессе найден новый устойчивый маршрут или ошибка в старых правилах.
