# Передача файлов через МСП

Playbook о том, как корректно отдавать большие артефакты (Markdown, отчёты, выгрузки) через MCP-коннектор, чтобы они реально доходили до владельца. Зафиксировано 2026-06-01 по итогам выгрузки Zoom-транскриптов.

## Проблема (симптомы)

1. **Файл «не доходит».** MCP-тул возвращал весь Markdown прямо в ответе. Клиент-коннектора (Claude.ai / Hermes) **режет длинные ответы**, поэтому полный документ не доезжал в чат, хотя тул отрабатывал верно.
2. **«Тысяча вызовов» вместо одного файла.** На запрос «выгрузи за несколько дат» ассистент пытался собрать `call_id` вручную: листал `list_zoom_calls` постранично (а список тяжёлый из-за длинных аналитических заметок → обрезка → пагинация по одному элементу) и зацикливался, так и не собрав общий файл.

## Корневые причины

- **Возврат большого контента инлайном** в результате tool-call — клиенты усекают, доставки нет.
- **Нет селектора под «набор конкретных дней».** Был только диапазон `date_from/date_to` или явные `call_ids`. Для несплошных дат (7, 8, 14, 15, …) модель вынужденно собирала id через листинг.
- **Описание/инструкция подталкивали к листингу** («если сомневаешься — посмотри список»), что усугубляло цикл.

## Рабочее решение (паттерн)

1. **Большой результат → сохранить на сервере и вернуть ССЫЛКУ, а не текст.**
   - Файл пишется в `exports/zoom/`; тул возвращает компактно `{download_url, filename, chars, bytes, preview}` (без полного текста; `preview` — первые строки для контекста).
   - Публичная ссылка без логина и **с истечением срока**: `GET /zoom-export/<expires_at>/<token>/<file>`, где `token = HMAC("<expires_at>:<filename>", FLASK_SECRET_KEY|MCP_SHARED_SECRET)` (стейтлес, неугадываемый). Маршрут в `AUTH_EXEMPT_PREFIXES`. Неверный токен или `now > expires_at` → 404; просроченный файл удаляется при обращении. TTL = `ZOOM_EXPORT_TTL_SECONDS` (по умолчанию 1800 = 30 мин).
   - **Чистка диска:** `cleanup_zoom_exports()` удаляет файлы старше TTL и вызывается на каждом новом экспорте; плюс cron `albery-zoom-export-cleanup` сметает старое каждые 10 мин. Так файлы не копятся.
   - Абсолютный URL строится из `MCP_HOST` (в прод-`.env` задан `MCP_HOST=mcp.m4s.ru`), с фолбэком на хост текущего запроса.
   - **Имя файла в URL — короткое и ASCII** (`<время>_<8hex>.<ext>`), человеческое название лежит рядом в файле `.name` и отдаётся заголовком `Content-Disposition` (`write_export_display_name` / `export_display_name`). Так было не всегда: русское название подставлялось в URL и раздувало его до ~470 символов percent-кодировки. Имя покрыто подписью, а модель обязана перенести ссылку в свой ответ — на длинной строке она роняет символы, и подпись перестаёт сходиться. Пользователь получал 404 с первой выдачи и рабочую ссылку со второй (20.07.2026, задача 1748; доказано пересчётом HMAC: токен сходился с ИСПРАВЛЕННЫМ именем).
   - **Ремонт ссылок на выходе:** `repair_export_links()` вызывается в `_b24_app_reply` для КАЖДОГО исходящего сообщения бота. Если подпись не сходится с именем файла — ссылка перестраивается: токен неподделываем, поэтому по нему однозначно находится файл, для которого он выдан (обратный случай — битый токен при живом файле — переподписывается). Адресовать можно только существующие файлы в `exports/zoom/`, новых прав это не даёт.
   - **Правило на будущее:** всё, что модель обязана дословно воспроизвести в ответе, должно быть КОРОТКИМ и ASCII. Длинная percent-кодированная строка внутри подписи — это гарантированные случайные 404.
2. **Дать селектор под реальные формулировки задачи.** Добавлен параметр `dates` (массив `YYYY-MM-DD`): один вызов собирает ВСЕ встречи перечисленных дней. Приоритет селекторов: `call_ids` > `dates` > `date_from/date_to`.
3. **Описание тула и AI-инструкция должны прямо запрещать обходные пути.** В описании `export_zoom_transcripts_markdown` и в AI-инструкции «Выгрузка транскриптов созвонов в Markdown»: для набора дней — `dates`, для сплошного периода — `date_from/date_to`; **не листать `list_zoom_calls` ради сбора id**; один запрос = один вызов = один файл; отдавать `download_url`.

## Где что лежит (конкретика)

- `app.py`: `export_zoom_call_markdown` / `export_zoom_calls_markdown` (ядро рендера, `dates`-селектор), `*_markdown_link` (сохранение + ссылка), `save_zoom_markdown_export`, `_zoom_export_token`, `_zoom_export_public_url`, маршрут `zoom_export_download` (`/zoom-export/<token>/<file>`), HTTP `/api/zoom-export.md` (`?dates=`, `?date_from=&date_to=`, `?ids=`), `ZOOM_EXPORT_DIR`, `AUTH_EXEMPT_PREFIXES`.
- `mcp/context_server.py`: тулы `export_zoom_call_markdown`, `export_zoom_transcripts_markdown` (возвращают ссылку; описания со строгими правилами выбора).
- AI-инструкция: `scripts/ai_instruction_zoom_export.md` → `scripts/upsert_albery_ai_instruction.py "Выгрузка транскриптов созвонов в Markdown" scripts/ai_instruction_zoom_export.md`.
- Прод-конфиг (не в git): `MCP_HOST=mcp.m4s.ru` в `/var/www/albery/.env`.

## Чек-лист «сделать так же в следующий раз»

- [ ] Артефакт может быть большим? → НЕ возвращать инлайном. Сохранить файл, вернуть `download_url` + `preview`.
- [ ] Ссылка должна работать без логина? → публичный маршрут с HMAC-токеном + добавить префикс в `AUTH_EXEMPT_PREFIXES`; абсолютный URL из `MCP_HOST`.
- [ ] Под какие формулировки нужен селектор? → дать прямой параметр (например список `dates`), чтобы не собирать id листингом.
- [ ] Прописать в описании тула И в AI-инструкции, какой селектор когда использовать и что листинг/ручной сбор id запрещены.
- [ ] После изменения набора/описаний MCP-тулов — перезапустить `albery` и `hermes-gateway`, попросить владельца `/reset` (или новый чат коннектора).

## Как проверено

- MCP `tools/call` возвращает `download_url` + `preview`, без полного текста; `curl -L` по ссылке → HTTP 200, `text/markdown`; неверный токен → 404.
- `export(dates=[8 дат])` → 11 встреч в одном файле, один вызов, без листинга.
