# О проекте Albery

Albery — управленческая AI/MCP-система для анализа работы компании по данным Bitrix, Zoom, чатов и корпоративных документов. Проект помогает владельцу и руководителям видеть рабочий контекст: задачи, созвоны, отчёты, риски, рекомендации, оценки руководителей и следующие действия.

Изначально проект вырос из Bitrix weekly export mini app, но текущая система значительно шире: backend, web UI, PostgreSQL, MCP-сервер, Hermes-оркестрация, AI-инструкции, Zoom/Bitrix/Google Drive-интеграции, OCR, отчёты, рекомендации и внешние действия в Bitrix.

## 1. Назначение

Albery нужен для управленческой аналитики и контроля исполнения:

- собирать рабочий контекст из Bitrix, Zoom, чатов и документов;
- превращать сырые сообщения, транскрипты и задачи в понятные отчёты;
- помогать владельцу видеть проблемы, динамику и рекомендации;
- превращать согласованные выводы в действия в Bitrix;
- давать Hermes/ИИ безопасный доменный доступ к данным через MCP;
- хранить живые AI-инструкции, по которым агент строит отчёты и отвечает владельцу.

## 2. Основные пользователи и роли

- Владелец / собственник — получает owner-отчёты, рекомендации, своды по руководителям и подтверждает внешние действия.
- Руководители — получают задачи, оценки по созвонам и операционные выводы.
- Участники созвонов и чатов — фигурируют в задачах, транскриптах, рекомендациях и отчётах.
- Hermes / AI-агент — читает контекст через MCP, строит отчёты и инициирует действия только по правилам подтверждения.
- Оператор/разработчик — поддерживает backend, MCP, UI, миграции, инструкции, cron и деплой.

## 3. Основные источники данных

- Bitrix:
  - задачи;
  - комментарии;
  - пользователи;
  - оргструктура;
  - события задач/команды;
  - отправка задач и сообщений.
- Zoom:
  - записи созвонов;
  - участники;
  - транскрипты;
  - аналитические заметки и AI-отчёты по созвонам.
- Чаты:
  - сообщения;
  - вложения;
  - OCR изображений/PDF;
  - дневные и недельные отчёты по чатам.
- Google Drive / Apps Script:
  - корпоративные документы;
  - зеркалирование файлов в базу/контекст;
  - webhook/sync-события.
- PostgreSQL:
  - основной источник состояния проекта;
  - отчёты, инструкции, события, индексы, связки, версии и результаты workflow.
- AI-инструкции:
  - живые правила агента из БД;
  - промпты/контракты в git;
  - runtime-настройки поведения Hermes.

## 4. Компоненты системы

### Backend

Главный backend — Flask-приложение. В нём сосредоточены:

- HTTP-маршруты;
- workflow-функции;
- интеграции Bitrix, Zoom, Google Drive;
- обработчики webhooks;
- отчёты;
- OCR;
- owner-рекомендации;
- AI-контракты;
- экспортные ссылки и файлы;
- отправки и действия в Bitrix.

Аудит показал, что backend сейчас является большим central/god-object. Поэтому следующие изменения должны быть узкими и проверяемыми; крупный рефакторинг допустим только после расширения safety-тестов.

### Frontend

Frontend — React/Vite web UI для просмотра и ручного управления:

- реестрами;
- отчётами;
- чатами;
- Zoom-созвонами;
- AI-инструкциями;
- настройками;
- ручными синхронизациями и workflow-действиями.

По аудиту frontend содержит много вызовов legacy `/api/*`. Поэтому UI нельзя считать отдельно безопасным: его нужно привести к тем же правилам risk class и confirmation policy, что MCP и backend.

### PostgreSQL

PostgreSQL хранит данные и состояние системы:

- задачи и комментарии;
- сообщения и транскрипты;
- Zoom calls и AI-отчёты;
- owner daily/weekly reports;
- рекомендации и события;
- AI-инструкции и версии;
- company knowledge;
- workflow-состояние.

Текущий миграционный стандарт: базовая схема выступает как v1, последующие миграции идут поверх неё. Перед изменениями production-схемы нужен отдельный preflight и проверка бэкапа.

### MCP

MCP — граница доступа Hermes/ИИ к проекту. Он предоставляет доменные инструменты вместо произвольного SQL или прямого доступа к внешним сервисам.

Важно: MCP не является чисто read-only. В нём есть:

- чтение контекста;
- внешнее чтение;
- локальные экспорты;
- OCR/workflow;
- записи отчётов и рекомендаций;
- изменение живых AI-инструкций;
- внешние действия в Bitrix.

Для внешних действий обязателен путь: просмотр → preview → явное подтверждение владельца → вызов с `confirm=true`.

### Hermes workflows

Hermes работает как Telegram-оператор и использует MCP-инструменты для:

- чтения live AI-инструкций;
- построения отчётов;
- подготовки owner-рекомендаций;
- согласования внешних действий;
- запуска scheduled/cron workflows;
- отправки владельцу превью и итогов.

Hermes не должен выполнять внешние действия только потому, что модель решила, что они полезны. Внешнее действие требует подтверждения владельца и server-side confirm-gate.

### Webhooks

Вебхуки Bitrix, Zoom и Google Drive — отдельный класс действий: `webhook_ingest_sync`.

Они не требуют owner-confirm, потому что являются автоматическим входящим ingest-потоком. Но для них обязательны:

- секрет/подпись;
- идемпотентность;
- журналирование;
- отсутствие лишних внешних действий без отдельного workflow.

## 5. Основные потоки данных

### Bitrix context flow

1. Bitrix tasks/users/comments/org structure подтягиваются в backend/БД.
2. Данные индексируются и становятся доступны через UI и MCP.
3. Hermes читает задачи и комментарии через MCP.
4. Если нужен внешний action, Hermes показывает preview владельцу.
5. После явного подтверждения вызывается инструмент с `confirm=true`.
6. Backend/MCP создаёт, удаляет или отправляет действие в Bitrix.

### Zoom analysis flow

1. Zoom recording/transcript попадает в систему через sync/webhook.
2. Backend сохраняет созвон, участников и transcript segments.
3. AI-контракт строит analytical note / report.
4. Hermes или UI показывают preview отчёта и возможной рассылки.
5. После подтверждения dispatch создаёт задачи/сообщения в Bitrix.

### Chat/OCR/report flow

1. Чаты и сообщения синхронизируются в PostgreSQL.
2. Вложения проходят OCR workflow при необходимости.
3. По сообщениям и OCR строятся дневные/недельные отчёты.
4. Owner reports используют эти источники как evidence.
5. Рекомендации сохраняются как версии/события и могут быть отправлены в Bitrix только после согласования.

### Company knowledge flow

1. Корпоративные документы попадают в Google Drive.
2. Apps Script / sync зеркалит их в контекст проекта.
3. MCP-инструменты `list_company_files`, `get_company_file`, `search_company_knowledge` дают агенту доступ к знаниям.
4. Ответы и отчёты должны ссылаться на факты из документов, а не на догадки.

### AI-instruction flow

1. В git лежат промпты/контракты и вспомогательные скрипты.
2. В БД живут активные AI-инструкции, которые Hermes читает перед работой.
3. Изменение AI-инструкции меняет runtime-поведение агента.
4. Поэтому AI-инструкции относятся к `db_write_current`, а не к обычной текстовой правке.
5. Для правки нужен preview, версия/журнал и понятный план отката.

## 6. Контуры доступа и записи

### Read-only

Можно без подтверждения:

- читать задачи, комментарии, отчёты, транскрипты, оргструктуру;
- искать сообщения;
- смотреть готовность отчётов;
- читать company knowledge;
- получать runtime status.

### Workflow / DB write

Нужен явный контекст, а для текущего состояния — preview:

- генерация и сохранение owner reports;
- сохранение Zoom reports;
- OCR processing;
- изменение рекомендаций;
- обновление AI-инструкций;
- создание/удаление текущих версий отчётов или промптов.

### External action

Всегда требуется owner-confirm и `confirm=true`:

- создание Bitrix tasks;
- удаление Bitrix tasks;
- dispatch Zoom operational tasks;
- dispatch leader evaluations digest;
- отправка owner recommendations;
- отправка PDF;
- личные сообщения в Bitrix.

## 7. Safety policy

Базовое правило Albery:

> Любое действие, которое меняет внешнюю систему или поведение живого агента, должно быть видно владельцу до выполнения и иметь техническую защиту от случайного вызова.

Практически это означает:

1. Сначала чтение и сбор evidence.
2. Потом preview точного результата: получатель, текст, задача, дедлайн, файлы, side effects.
3. Потом явное согласие владельца.
4. Потом технический вызов с `confirm=true`.
5. Потом отчёт о результате.

## 8. Надстройка: оценка руководителей по Zoom-созвонам

### Зачем

По каждому созвону, помимо выделения задач, нужно:

1. оценивать, как руководители ведут созвон — держат ли планку, соответствует ли встреча регламенту, не уходит ли разговор в сторону;
2. отправлять каждому участнику краткую выжимку созвона, руководителям — ещё и персональную оценку, а тем, у кого есть задачи, — список задач;
3. раз в неделю собирать собственнику свод по руководителям.

### Как это работает

```text
Zoom-созвон
  → транскрипт
  → AI-отчёт по контракту zoom_processing
  → сохранение analytical note / AI report
  → Hermes показывает владельцу полный отчёт и preview рассылки
  → после подтверждения dispatch создаёт задачи в Bitrix
  → раз в неделю Hermes собирает свод руководителей
```

### Контракт отчёта

Источник правды в git:

- `scripts/zoom_processing_prompt_v9.md`;
- `scripts/update_zoom_processing_prompt_leader_eval.py`.

Ключевые поля:

- `dispatch_summary` — короткая выжимка для рассылки;
- `is_leader` и `role_on_call` у участников;
- `leader_evaluations` — оценки руководителей;
- правила назначения ответственного;
- подсветка unmatched participants/assignees на согласовании.

### Dispatch в Bitrix

Диспетчер берёт сохранённый отчёт и создаёт агрегированные задачи по участникам. Совместимость со старыми отчётами сохраняется: старые отчёты раздаются без новых блоков оценки.

### Свод собственнику

Свод руководителей агрегирует оценки за период и создаёт одну задачу владельцу только после согласования.

## 10. Что требует следующей стандартизации

1. Добавить machine-readable risk metadata для MCP-инструментов.
2. Проверить все external-action tools на обязательный confirm-gate.
3. Решить судьбу legacy API и UI: поддерживаемый admin UI или legacy shell.
4. Добавить отдельные playbooks для deploy, AI-инструкций, Hermes cron и migration workflow.
5. Постепенно выделять домены из крупных backend/frontend/MCP-файлов только после тестовой защиты.

## 11. Что нельзя делать без отдельного решения

- Переписывать backend целиком.
- Удалять legacy API без проверки реального production/UI режима.
- Запускать миграции по production-БД без backup-check и preflight.
- Менять живые AI-инструкции без preview, версии и плана отката.
- Менять Hermes cron/промпты без отдельного playbook.
- Считать frontend неиспользуемым только потому, что часть API помечена legacy.
