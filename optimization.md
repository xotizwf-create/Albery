# Оптимизация MCP-сервера и инструкций для ИИ

Документ фиксирует, что и как было улучшено в MCP-слое проекта (`mcp/context_server.py`,
инструкции в `ai_instruction_folders` и `mcp/instructions/*.md`), чтобы ИИ
**быстрее ориентировался в системе** и **быстрее делал отчёты**.

Цель: уменьшить число round-trip'ов к инструментам и объём токенов в ответах, не ломая
действующую систему рекомендаций по работе с инструментами.

---

## 1. Как устроены инструкции (модель до изменений)

ИИ читает инструкции с трёх уровней:

1. **Глобальные правила** — таблица `ai_instruction_folders`, отдаются «вживую» через
   `load_ai_instructions()`.
2. **Навигация** — статичный код `tool_get_context_guide` (карта источников + воркфлоу).
3. **Контракты отчётов** — таблица `ai_prompts`, через `get_report_contract`.
4. Плюс `mcp/instructions/*.md` — «исходник» для копирования в UI.

В полном коннекторе **40 инструментов** (стало 41).

---

## 2. Найденные узкие места

### 2.1. Ориентация ИИ
- **Тройная загрузка всех инструкций.** `start_here_always_read_ai_instructions`,
  `get_context_guide` и `get_ai_instructions` — все возвращали полное дерево инструкций.
  Полный текст инструкций ≈ **67 000 символов (~17k токенов)** приезжал в контекст 2–3 раза.
- **Нет маршрутизации по задаче.** `get_context_guide` отдавал все воркфлоу и все правила
  сразу; `load_ai_instructions` — всё дерево. Нельзя было запросить «только инструкцию под owner daily».
- **Битое имя инструмента.** В `.md` было `get_ai_prompts` (такого инструмента нет; реальный — `get_report_contract`).
- **Налог на манифест.** Длинный `TOOL_USAGE_CONTRACT` приклеивался к описанию каждого из 40 инструментов.

### 2.2. Скорость отчётов
- **Готовность отчёта собиралась вручную.** `get_period_index`/`get_compact_export` не сообщали,
  по каким чатам уже есть daily-отчёт, по каким Zoom есть `analytical_note`, есть ли вчерашний
  owner-отчёт. Это десятки пробных вызовов на owner/weekly отчёт.

### 2.3. Поиск задач (главная причина «5 минут» и «очень долго»)
Измерено на реальной БД (3456 задач) — **БД не виновата**:

| Запрос | План | Время |
|---|---|---|
| Поиск по `bitrix_task_id = X` | Index Only Scan (UNIQUE) | **0.6 мс** |
| По человеку (`responsible_bitrix_user_id`) | Seq Scan (мелкая таблица) | 1.8 мс |
| По периоду (`COALESCE(...)::date`) | Seq Scan | 1.7 мс |

Реальные причины медленности:
- **В `search_tasks` не было параметра `bitrix_task_id`.** Поиск «задачи 318241» шёл через
  `query` → `title/description ILIKE '%318241%'` (номера там нет → 0 совпадений), после чего ИИ
  листал и сканировал огромные ответы. Отсюда 5 минут.
- **Гигантские payload'ы** (задержка модели растёт пропорционально объёму, который она читает):

| Ответ | Размер |
|---|---|
| `search_tasks` за 30 дней (200 задач) | ~214 800 символов (**~53 700 токенов**), половина — полные `description` |
| `get_compact_export` за неделю | ~538 600 символов (**~134 600 токенов**) |
| └ из них `company_profile` (вся база «О компании») | ~198 000 символов (~50k токенов) |
| └ из них owner-отчёты (`report_text` + `raw_json`) | ~161 700 символов |

### 2.4. Текст инструкций
- **Противоречие про `no_data`.** Папка «Генерация отчётов по чатам» велела для пустого дня
  сохранять отчёт `no_data`, а профильные инструкции — пропускать пустой день и `no_data` не создавать.
  (На проде это уже было исправлено; старая версия осталась только в локальной dev-БД.)
- **Нет правила поиска по номеру задачи** ни в «Bitrix-задачи», ни в классификаторе.

## 3. Что сделано

### 3.1. Ориентация ИИ (`mcp/context_server.py`)
- `start_here_always_read_ai_instructions` отдаёт полный текст инструкций **один раз**
  (обязательное чтение); `next_tool_guidance` обновлён.
- `get_context_guide` больше **не дублирует** инструкции — возвращает компактный
  `ai_instructions_index` (пути + размеры) и подсказку.
- `get_context_guide(intent=…)` — маршрутизация по задаче: только нужный воркфлоу и
  релевантные источники. Без `intent` — полный гид (обратная совместимость). Есть алиасы интентов.
- `get_ai_instructions(path=…)` — точечное дочитывание одной папки по пути.
- `TOOL_USAGE_CONTRACT` сокращён до короткого указателя.
- В `.md` исправлено `get_ai_prompts` → `get_report_contract`.

### 3.2. Готовность отчётов
- Новый инструмент **`get_report_readiness(date_from, date_to)`**. За один вызов по каждому дню:
  активные чаты с сообщениями и какие уже имеют daily-отчёт (`missing_daily_reports`),
  Zoom-созвоны и какие уже имеют `analytical_note` (`missing_zoom_reports`),
  есть ли текущий/предыдущий owner-отчёт, флаг `ready_for_owner_daily`.
- Воркфлоу `daily/weekly/owner` в `get_context_guide` и инструкции переписаны на вызов этого агрегата
  вместо поштучных проверок.

### 3.3. Поиск задач и размер ответов
- `search_tasks` принимает **`bitrix_task_id`** — мгновенный поиск по UNIQUE-индексу
  (одна задача). Описание инструмента подсказывает: номер задачи → `bitrix_task_id`, не `query`.
- `search_tasks` принимает **`include_full_description`**; по умолчанию `description` обрезается
  до превью 500 символов + флаг `description_truncated`/`description_full_length`.
- `get_compact_export`: профиль компании → лёгкий индекс папок (тела документов читаются через
  `search_company_knowledge`/`get_company_file`); owner-отчёты → только summary/риски/рекомендации;
  лимиты снижены.

### 3.4. Текст инструкций (`scripts/update_chat_report_task_instructions.py`)
Идемпотентный скрипт, применённый к прод-БД (и локальной):
- **Правило поиска по номеру задачи** добавлено в «Bitrix-задачи» и в маршрут «Задача/срок»
  классификатора (это правки `required` — обязательные).
- **Правило пустого дня** (пропускать день, `no_data` не создавать) — best-effort: если правило
  уже есть в другой формулировке, скрипт его распознаёт и ничего не перезаписывает.

## 4. Результат (замеры до/после)

| Сценарий | Было | Стало |
|---|---|---|
| Поиск задачи по номеру | ~5 мин блужданий, скан 54k-токенных ответов | **1 вызов, ~150 токенов, ~40 мс** |
| `get_compact_export` (неделя) | ~134 600 токенов | **~36 700 токенов (−73%)** |
| `search_tasks` за период | полный `description` в каждой строке | превью 500 симв. + `include_full_description` по запросу |
| Навигация ИИ | инструкции грузились 2–3 раза (~17k токенов ×2–3) | один раз; маршрутизация `intent=` / `path=` |
| Готовность отчёта | десятки пробных вызовов | один `get_report_readiness` |

Проверено на проде: инструментов 41, `get_report_readiness` есть, `health: ok`,
поиск `bitrix_task_id` возвращает одну задачу мгновенно, правило by-id в живых инструкциях.

---

## 5. Что НЕ делалось и почему
- **Индексы на `responsible_bitrix_user_id` / выражение даты** — не нужны: на 3456 задачах
  seq scan занимает ~2 мс. Имеет смысл при ~100k+ строк.
- **Единый источник правды (сидинг `ai_instruction_folders` из `.md`)** и **кэш `load_ai_instructions`
  по `updated_at`** — отложены (опционально, не влияют на текущую скорость).

---

## 6. Деплой
- Код: ветка `feature/mcp-speed-instructions` → `main` (`1ffd24b`), запушено в GitHub.
- Сервер: `./scripts/update_server.sh` (git pull --ff-only + build + `systemctl restart albery`).
  Flask на `:5002` обслуживает и MCP, поэтому рестарт подхватывает изменения `mcp/context_server.py`.
- Инструкции на проде: `.venv/bin/python scripts/update_chat_report_task_instructions.py`
  (изменения в `ai_instruction_folders` вступают в силу сразу, без деплоя).
