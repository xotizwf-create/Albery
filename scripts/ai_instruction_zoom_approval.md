# Что делать на «ставь» после zoom-to-tasks сводки

КОГДА владелец отвечает в Telegram «ставь» / «создавай» / «да» / «создаём» / «отправляй» / «ок» / «давай» **после короткой сводки от cron `zoom-to-tasks`** (где был список задач созвона и финал «Создаём задачи в Битрикс?») — сразу выполни эту цепочку и **НЕ задавай уточнений**:

1. Вызови `albery list_pending_zoom_operational_dispatches()` без аргументов — он вернёт массив `pending` с `call_id`, `call_date`, `topic`, `start_time_msk` ТОЛЬКО за сегодня (Europe/Moscow). Это именно те созвоны которые владелец видел в недавней сводке и одобрил. Не передавай date_from/date_to явно — старые непереданные созвоны (за предыдущие дни) намеренно скрыты, чтобы не отправить лишнее.
2. Если `pending_count == 0` — ответь владельцу: «Все обработанные созвоны уже отправлены в Битрикс.» и закончи.
3. Если `pending_count >= 1` — для КАЖДОГО элемента вызови `albery dispatch_zoom_operational_tasks call_id=<call_id> confirm=true`. Этот инструмент сам сгруппирует `operational_tasks` по ответственному, создаст **одну агрегированную задачу** «Итоги созвона ЧЧ:ММ» на каждого человека, дедлайн = дата созвона 19:00 МСК, описание = стандартный шаблон UI Albery («Ознакомьтесь со списком выделенных из созвона задач и поставьте себе самые важные в Битрикс…»).
4. После всех `dispatch_zoom_operational_tasks` верни владельцу короткий человеческий ответ без технических id:
   ```
   Создано M задач в Битрикс по созвону <тема, ДД.ММ.ГГГГ>:
   — Артур Степанян (1 задача)
   — Наталья Викторовна Горюнова (1 задача)
   Дедлайн ДД.ММ.ГГГГ 19:00 МСК.
   ```

## Что НЕЛЬЗЯ делать на «ставь» после zoom-сводки

- НЕ переспрашивай владельца «напишите название/ответственного/дедлайн» — у `dispatch_zoom_operational_tasks` все обязательные поля собираются автоматически из сохранённого отчёта по `call_id`.
- НЕ вызывай `create_bitrix_task` для отдельных задач — это создаст много мелких задач вместо одной агрегированной, формат не совпадёт с Albery UI «Отправка задач».
- НЕ ищи `call_id` вручную через `list_zoom_calls` / `get_zoom_call_transcript` — `list_pending_zoom_operational_dispatches` уже отфильтровал именно те созвоны, по которым нужна отправка.
- НЕ читай `get_org_structure`, `get_zoom_call_transcript`, `get_report_contract` — `dispatch_zoom_operational_tasks` сам всё подтянет из БД, сэкономь токены.
- НЕ показывай список задач владельцу повторно — он уже видел их в исходной сводке от cron и одобрил.

## Другие ответы владельца

- «Не ставь» / «не надо» / «отмена» / «нет» — ничего в Битриксе не создавай. Ответь: «Понял, задачи не ставлю.» и закончи.
- «Правки по №N: <текст>» — режим редактирования отдельной задачи пока не автоматизирован. Ответь: «Правки по конкретной задаче я пока не умею перезаписывать в сохранённом отчёте. Опиши пожалуйста, что именно поменять, я учту устно и предложу обновлённый текст перед отправкой.» и НЕ вызывай `dispatch_zoom_operational_tasks`.

## Признаки контекста (как понять что речь идёт именно про zoom-to-tasks)

Сразу выше в этой же Telegram-сессии было сообщение от cron с шаблоном:
```
Созвон: ДД.ММ.ГГГГ — <тема>
Предлагаю поставить задачи в Битрикс:
1. ФИО — текст. Срок: ... Критерий: ...
2. ...
Создаём задачи в Битрикс? Ответь «ставь» — я по каждому ответственному создам одну агрегированную задачу...
```

Если такого сообщения в недавнем контексте не было, и владелец просто пишет «ставь задачу» / «поставь задачу <Х>» — это другой сценарий (создание единичной задачи), используй `create_bitrix_task` со всеми стандартными проверками.
